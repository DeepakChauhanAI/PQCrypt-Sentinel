NIST
US · Primary Authority
FIPS 203 — ML-KEM (Module-Lattice Key Encapsulation Mechanism)
Finalized Aug 2024. Replaces RSA/ECDH for key exchange. Formerly CRYSTALS-Kyber. Ready for immediate implementation.
Key Points Specifies ML-KEM, a key-encapsulation mechanism for establishing shared secret keys over a public channel. Security is based on the hardness of the Module Learning with Errors (MLWE) problem, which is believed to be resistant to quantum attacks. Defines three parameter sets — ML-KEM-512, ML-KEM-768, and ML-KEM-1024 — in increasing order of security strength and decreasing performance. The shared secret established by ML-KEM is then used with symmetric-key algorithms for encryption and authentication. This is the primary replacement for RSA and ECDH key exchange in TLS, SSH, and other protocols. Immediate implementation is recommended.
2024STANDARD
FIPS 204 — ML-DSA (Module-Lattice Digital Signature Algorithm)
Finalized Aug 2024. Replaces RSA/ECDSA for digital signatures. Formerly CRYSTALS-Dilithium.
Key Points Specifies ML-DSA, the primary post-quantum digital signature algorithm based on the CRYSTALS-Dilithium scheme. Security is grounded in the hardness of lattice problems (MLWE and SelfTargetMSIS). Defines three parameter sets — ML-DSA-44, ML-DSA-65, ML-DSA-87 — offering different security/performance trade-offs. Used for signing code, firmware, documents, certificates, and authenticating TLS connections. Replaces RSA-PSS and ECDSA signatures across all banking and payment infrastructure. Notably larger signature sizes than RSA/ECDSA — important for scanning findings on protocols with size constraints (e.g., DNS, some IoT protocols).
2024STANDARD
FIPS 205 — SLH-DSA (Stateless Hash-Based Digital Signature)
Finalized Aug 2024. Additional signature scheme using hash-based construction. Backup signature algorithm.
Key Points Specifies SLH-DSA (based on SPHINCS+), a stateless hash-based signature scheme. Its security relies only on the properties of hash functions — a fundamentally different and more conservative mathematical assumption than ML-DSA. Considered the "backup" signature algorithm if lattice-based schemes are ever broken. Defines 12 parameter sets across small (fast signing, large signatures) and fast (smaller signatures, slower signing) variants. Particularly suited for long-lived signatures such as root CA certificates, firmware signing keys, and code-signing infrastructure where the signing key must remain valid for 10–20+ years. Higher performance cost than ML-DSA but stronger security guarantees due to its different mathematical foundation.
2024STANDARD
NIST SP 1800-38B — Cryptographic Discovery
Automated discovery tools and methodology for identifying quantum-vulnerable cryptography. Lessons learned from real discovery exercises. Priority #1 for this project.
Key Points Covers three discovery domains: (1) Code pipeline — scanning source code, binaries, and dependencies in CI/CD for vulnerable algorithms via SAST tools; (2) Operational systems — filesystem scanning for cryptographic libraries, keystores (PKCS#12, JKS), certificate files (.pem, .p12, .pfx), and application binaries; (3) Network/transport — passive and active scanning of TLS, SSH, and IPsec traffic for cipher suites and key exchange parameters. Introduces the Cryptography Bill of Materials (CBOM) concept using CycloneDX format to create a machine-readable inventory of every crypto asset. Defines a normalized output format (SARIF for static analysis, JSON/CBOM for network) so findings from multiple tools can be aggregated into a single risk dashboard. Identifies the vulnerable algorithms to flag: RSA, ECDSA, ECDH, DH, DSA, EdDSA, MQV. Emphasizes that most organizations cannot currently see all cryptography in use — the first step is building visibility, then prioritizing replacement using a risk-based methodology.
2024DISCOVERY
NIST SP 1800-38A — Executive Summary & Preparation Overview
Overview of the migration project scope, outcomes, and organizational preparation guidance.
Key Points Executive-level overview of the entire NIST NCCoE PQC Migration project. Sets out four mandatory steps every organization must take: (1) Establish a Quantum Readiness Roadmap; (2) Prepare a Cryptographic Inventory; (3) Discuss quantum-safe roadmaps with technology vendors; (4) Determine supply chain quantum-readiness. Explains that previous cryptographic transitions (3DES→AES, SHA-1→SHA-2, RSA→ECC) each took years or decades, and the PQC transition will be more complex due to the scale and depth of deployment. Emphasizes that the "harvest now, decrypt later" (HNDL) threat means migration cannot wait for quantum computers to arrive — data being captured today is already at risk. Intended for CISOs, CIOs, and board-level stakeholders to understand the scope and urgency.
2023OVERVIEW
NIST SP 1800-38C — Interoperability & Performance Testing
Results from testing PQC algorithm implementations for interoperability and performance. Relevant for assessing migration feasibility across banking systems.
Key Points Documents real-world testing of PQC algorithm implementations across TLS, SSH, and X.509 certificate use cases. Identifies compatibility issues encountered when mixing classical and PQC implementations across different vendors and platforms. Key findings: ML-KEM and ML-DSA introduce larger key and signature sizes that can break assumptions in existing protocols (e.g., TLS handshake size limits, DNSSEC packet sizes). Performance testing shows acceptable overhead for most enterprise use cases, but legacy hardware (ATM controllers, HSMs) may require firmware updates or replacement. Provides a basis for assessing which systems in a bank's stack can be migrated quickly vs. which require hardware replacement or protocol redesign.
2023TESTING
NIST IR 8547 — Transition to PQC Standards
Federal transition planning and algorithm selection. Defines deprecation: RSA/ECC disallowed after 2030 (112-bit) and 2035 (all). Core compliance reference for scanner output.
Key Points Defines NIST's official deprecation and disallowance timeline for quantum-vulnerable algorithms — this is the compliance backbone for every finding your scanner produces. Key dates: RSA and ECC with 112-bit security disallowed after 2030; all remaining RSA/ECC/DH disallowed after 2035. FIPS 203, 204, and 205 are the designated replacements. Identifies which existing NIST standards (SP 800-56A/B/C, FIPS 186-5, SP 800-131A) are being deprecated and what replaces each one. Intended to align federal agencies, vendors, and critical infrastructure operators on a common transition schedule. For your scanner: every finding should be cross-referenced against this timeline to classify urgency as "disallowed now," "disallowed by 2030," or "disallowed by 2035."
Nov 2024TRANSITION
NIST CSWP 39 — Considerations for Achieving Crypto Agility
In-depth survey of approaches to crypto agility. Defines what it means to build systems that can swap algorithms without disruption. Shapes remediation recommendation output.
Key Points Defines crypto agility as the capability to replace and adapt cryptographic algorithms in protocols, applications, hardware, firmware, and infrastructure while preserving security and ongoing operations. Covers four areas: (1) Protocol-level agility — mandatory algorithm negotiation, hybrid schemes (classical+PQC in parallel), backward compatibility design; (2) Implementation-level agility — use of crypto APIs that abstract algorithm choice from application logic, hardware-agnostic crypto modules; (3) Organizational strategy — a five-phase plan covering inventory, risk assessment, prioritization, migration execution, and validation; (4) Future considerations — metrics for measuring agility, sector-specific guidance gaps. Key design principle: applications should call a crypto API layer that can be updated centrally, rather than hardcoding algorithm choices. Directly shapes what your scanner's remediation recommendations should look like — flag hardcoded algorithm calls, missing API abstraction layers, and non-agile certificate management.
2024AGILITY
NIST SP 800-208 — Stateful Hash-Based Signatures (LMS, XMSS)
Guidance on LMS and XMSS — hash-based signature schemes. Relevant for code signing, firmware signing in banking infrastructure.
Key Points Specifies LMS (Leighton-Micali Signature) and XMSS (eXtended Merkle Signature Scheme) — stateful hash-based signature schemes considered quantum-safe today. Unlike ML-DSA (stateless), these schemes maintain a state counter that must never repeat, making them suitable for closed, controlled signing environments such as firmware signing servers, HSM root keys, and code-signing infrastructure. Not appropriate for general-purpose TLS or interactive authentication. For banking: relevant for ATM firmware signing keys, core banking software update pipelines, and HSM key ceremonies. Scanner should identify where LMS/XMSS could replace RSA in code-signing flows.

CISA PQC Roadmap
Critical infrastructure guidance requiring owners/operators to begin PQC migration planning now. Defines "cryptographic inventory" as a mandatory first step.
Key Points CISA's guidance explicitly targets critical infrastructure sectors including financial services. Defines cryptographic inventory as Step 1 — mandatory before any other migration action. 2023–ongoingMANDATE
NSA CNSA 2.0 — Commercial National Security Algorithm Suite
Most aggressive US timeline. Mandates ML-KEM, ML-DSA, LMS/XMSS for national security systems. Sets the ceiling for what banking regulators may eventually require.
Key Points Sets the most aggressive migration timeline in any US government document. For national security systems: software and firmware signing must use LMS or XMSS by 2025; networking equipment must support ML-KEM by 2026; operating systems must support ML-KEM by 2027; all new products must be PQC-only (no classical) by 2030. While this directly applies only to NSS (National Security Systems), it signals the direction of travel for commercial regulation. Key implication for banking: any bank supplying technology to US government or defense customers must meet CNSA 2.0 timelines. Also sets the benchmark for what "fully quantum-safe" means — pure PQC, not hybrid — which is the long-term target even if hybrid is the acceptable transition state.
Updated Dec 2024NSS MANDATE
OMB M-23-02
Federal agency mandate for PQC inventory completion and migration planning. Directly shaped the NIST SP 1800-38 series.
Key Points White House memorandum that made PQC migration a mandatory federal government activity. Required all federal agencies to: (1) complete an inventory of quantum-vulnerable cryptographic systems within 90 days of issuance; (2) develop prioritized migration plans; (3) include PQC migration as a line item in IT budget requests. This memo directly commissioned the NIST SP 1800-38 series. For private sector banks: while not directly binding, this memo created the regulatory framework that CISA and banking regulators are now building on. It defines the language and structure ("cryptographic inventory," "migration plan," "prioritized systems") that regulators will use in future banking-specific guidance.

Banking & Payments Specific
FS-ISAC — PQC Future State Technical Paper
Phase 1 Discovery, risk exposure evaluation, risk scoring methodology and remediation guidance. Built specifically for financial institutions. Priority #2 for this project.
Key Points Five-phase risk assessment framework tailored for banks: Phase 1 (Discovery) — create a full cryptographic inventory using self-identification, automated scanning, SBOM/CBOM tools, and manual code review; Phase 2 (Risk Modeling) — score each asset using a Cryptographic Agility Index (CAI) that factors in whether the asset is internally built or from a third-party vendor, since vendor assets introduce additional supply chain risk; Phase 3 (Risk Tolerance) — define acceptable residual risk; Phase 4 (Prioritization) — rank assets for migration with HNDL-exposed data and internet-facing systems first; Phase 5 (Outcome) — vendor assessment and remediation roadmap. Highlights SWIFT as a sector-specific system requiring careful migration risk calculation. Discusses QKD (Quantum Key Distribution) as a supplementary measure. Defines three "degrees of readiness" for measuring progress: Inventory Complete → Migration Planned → Migration Executed.
2023BANKING
FS-ISAC — Preparing for a Post-Quantum World by Managing Cryptographic Risk
Risk management framework for financial institutions. Covers risk model application and remediation planning.
Key Points Applies Mosca's Theorem to financial institutions: if the time to migrate (y) plus the time data needs to remain secure (x) exceeds the time before a cryptographically relevant quantum computer arrives (z), action must begin now. Distinguishes three data exposure categories: data-in-motion (TLS, SWIFT), data-at-rest (encrypted databases, backup tapes), and data-in-use (keys in memory, HSM operations). The HNDL threat applies most acutely to long-lived data such as mortgage records, trade history, and customer PII — these need to be prioritized in the scanner's risk output. Provides practical guidance on building a cryptographic inventory in financial institutions where shadow IT and legacy systems are common. Discusses how to assess vendor PQC readiness using questionnaires and contractual audit rights.
Mar 2023BANKING
FS-ISAC — Building Cryptographic Agility in the Financial Sector
Evaluate, inventory, plan, validate, and execute framework for financial sector crypto agility. Priority #7.
Key Points Defines crypto agility for financial institutions as "the ability to adapt cryptographic solutions or algorithms quickly and efficiently in response to developments in cryptanalysis, emerging threats, or vulnerabilities — with no significant architectural changes and minimal business disruption." Core prerequisite: you cannot be crypto agile without a current, accurate cryptographic inventory — the scanner is the foundation. Provides nine core elements of a successful transition including topology design for a "crypto-as-a-service" central architecture, technical details of algorithm replacement at the API layer, and transition governance. Includes specific API schema designs for abstracting cryptographic operations in financial middleware. Critically warns that future PQC algorithms will likely have shorter operational lives than RSA — organizations must build for ongoing, repeatable transitions, not a one-off migration. Addresses the Shor and Grover algorithm threats in an appendix for executive education.

CMORG — Guidance for Post-Quantum Cryptography (TLP:CLEAR)
UK Capital Markets specific. Covers SWIFT, HSM, settlement scanning. Vendor PQC assessment guidance. Priority #6.
Key Points Explicitly states that banks as critical national infrastructure (CNI) must begin PQC uplift immediately — waiting is not an option given the scope of effort required. Defines a four-phase CMORG PQC Roadmap: (1) Discovery and Inventory — establish cryptographic inventory and identify dependencies; (2) Risk Assessment — prioritize assets, focusing on long-lived sensitive data and HNDL exposure; (3) Migration Planning — develop detailed roadmaps with vendor timelines; (4) Implementation — phased deployment of quantum-safe solutions with testing. Strongly emphasizes vendor readiness assessment: financial institutions must engage HSM vendors (Thales, Entrust, Utimaco), network providers, and software vendors to confirm their PQC migration timelines before planning internal migrations. Notes that AES symmetric encryption is considered low risk (Grover's algorithm requires doubling key length, which is manageable), while all public-key cryptography requires replacement. Aligns with NCSC three-phase timeline: 2028 → 2031 → 2035.
Apr 2025MARKETS
G7 Cyber Expert Group — Financial Sector PQC Roadmap
Global coordination framework for banking sector with 2030–2035 target window. Emphasises cryptographic agility as a core requirement.
Key Points Establishes the multilateral coordination framework for PQC migration across G7 financial systems (US, UK, EU, Canada, Japan, Italy, France). Key outcomes: all G7 central banks and financial regulators have agreed to a 2030–2035 target window for completing PQC migration in systemically important financial institutions. Cryptographic agility is a core requirement — institutions must not implement static PQC solutions that cannot be updated. Cross-border interoperability is a priority — correspondent banking, SWIFT, and payment network migrations must be coordinated to avoid breaking connectivity between institutions in different jurisdictions. Provides the political and regulatory context for why banks cannot delay — regulatory pressure from multiple directions is building simultaneously.
2024–2025G7
Payments & PQC — HSM Landscape Analysis
Detailed analysis of HSM vendor PQC readiness (Thales, Entrust, Utimaco, Futurex). FIPS 140-3 validation status as of mid-2025. Critical for HSM scanning layer.
Key Points Current PQC status of major HSM vendors as of early 2026: Thales Luna Network HSM 7 supports ML-KEM and ML-DSA from firmware 7.9+; Entrust nShield 5 supports from firmware 13.8+; Futurex achieved the first PCI HSM-validated PQC implementation (June 2025) covering PIN encryption with AES-256; Utimaco has PQC support in SecurityServer Se-Series. Critical finding: most HSM deployments in banks are running older firmware and will require scheduled maintenance windows for updates — some older hardware cannot be updated and will require physical replacement. BSI TR-02102-1 — Technical Guideline: Cryptographic Mechanisms
Most technically detailed on hybrid key derivation, minimum key sizes, approved cipher combinations, and legacy algorithm coexistence during migration. Priority #5.
Key Points Germany's authoritative technical specification for cryptographic algorithm selection, updated annually. Current version (2026-01) sets a minimum security level of 120 bits for all new systems — higher than the US 112-bit baseline. Key recommendations: (1) Symmetric encryption — AES-256 required for long-term data protection; AES-128 still acceptable for short-term; (2) Asymmetric/PQC — recommends ML-KEM and ML-DSA immediately; strongly recommends hybrid mode (classical+PQC simultaneously) during the transition period rather than replacing classical algorithms outright; (3) Key derivation — specifies approved HKDF constructions for hybrid key combination; (4) Hash functions — SHA-256 minimum, SHA-384/512 for long-term protection; (5) Legacy coexistence — provides explicit guidance on running classical and PQC schemes in parallel without weakening either. Protocol-specific recommendations are in companion documents TR-02102-2 (TLS), TR-02102-3 (IPsec), TR-02102-4 (SSH). Updated annually — check for current version before implementation decisions.
Updated 2024TECHNICAL
ENISA — Post-Quantum Cryptography Integration Study
EU-wide integration study covering protocol-level migration challenges across infrastructure types.
Key Points Analyzes PQC integration challenges across five protocol categories: TLS/HTTPS, SSH, S/MIME, IKE/IPsec, and PKI/X.509. Key findings: larger PQC key and signature sizes create performance issues in TLS handshakes and certificate chains; hybrid schemes are the recommended transition mechanism to maintain backward compatibility; PKI infrastructure (CAs, OCSP, CRLs) requires update before dependent systems can migrate. Provides EU-specific context on NIS2 compliance obligations and how PQC readiness maps to critical infrastructure requirements. Highlights that IoT and embedded systems are the hardest migration targets due to memory and compute constraints — relevant for ATM and branch hardware scanning.
Oct 2022EU
ECCG — Agreed Cryptographic Mechanisms Version 2
May 2025. Guidance for developers and decision-makers; recommends hybrid classical+PQC approach. Endorsed by EU national cybersecurity certification authorities.
Key Points Endorsed by all EU member state national cybersecurity certification authorities (NCCAs). Formally adopts the hybrid approach as the EU standard for the transition period — meaning any EU-regulated financial institution should be implementing hybrid, not pure PQC. Recommends ML-KEM and ML-DSA as the primary PQC algorithms. Provides guidance for software developers on correctly implementing hybrid key exchange (combining X25519 with ML-KEM-768) and hybrid signatures (combining ECDSA with ML-DSA). Particularly relevant for cross-border EU banking operations and compliance with DORA (Digital Operational Resilience Act), which is increasingly referenced in PQC planning discussions.

NCSC UK — PQC Migration Roadmap
Three-phase timeline: Discover & Plan (by 2028) → Prioritize & Pilot (2028–2031) → Complete Adoption (2031–2035). Centrally-managed migration guidance.
Key Points Sets the authoritative UK government timeline that UK-regulated banks must follow: Phase 1 by 2028 (complete cryptographic inventory and migration plan), Phase 2 by 2031 (migrate high-priority systems, run pilots), Phase 3 by 2035 (full adoption). Aug 2024UK
ANSSI — Views on the PQC Transition
French position: conservative approach, recommends hybrid schemes, detailed on algorithm selection rationale.
Key Points France's national cybersecurity agency takes a notably conservative stance: recommends hybrid classical+PQC as mandatory (not optional) during the transition, because the long-term security of the new PQC algorithms has not yet been fully stress-tested in practice. Does not recommend deploying standalone PQC-only implementations yet. Provides a detailed technical rationale for why each of the NIST-selected algorithms was chosen and discusses remaining open questions about their security. Discusses migration challenges for protocols like TLS 1.3, IKEv2, and S/MIME. Relevant for understanding the rationale behind hybrid scheme requirements — useful when explaining your scanner's recommendation to implement hybrid rather than pure PQC.
2023FRANCE
ETSI TR 103 619 — Migration Strategies to Quantum Safe Schemes
Migration strategy framework covering transition paths, hybrid approaches, and deployment considerations.
Key Points Provides a structured taxonomy of migration strategies: (1) Immediate replacement — swap vulnerable algorithm for PQC equivalent; (2) Hybrid mode — run classical and PQC in parallel; (3) Crypto agility — build systems that support algorithm negotiation and can switch without redeployment. Analyzes the trade-offs of each approach for different system types. Particularly useful for understanding how to categorize scanner findings by recommended remediation path — not every finding has the same fix. Covers deployment considerations for TLS, IPsec, and certificate infrastructure. Note: this is the 2020 version; ETSI has published more recent PQC standards (TR 103 949, 2025 key exchange standard) that supersede parts of this document.

Key Points Compares the leading commercial cryptographic discovery tools across three scanning modalities: (1) Network analyzers (SandboxAQ AQtive Guard, Cisco) — passive packet capture to fingerprint TLS/SSH cipher suites; (2) Certificate/PKI managers (Keyfactor, Venafi, DigiCert) — discover and track X.509 certificates across the enterprise including expiry, algorithm, and key size; (3) Application/filesystem analyzers (IBM, InfoSec Global, ISARA) — scan source code, binaries, and cryptographic libraries for algorithm usage.
