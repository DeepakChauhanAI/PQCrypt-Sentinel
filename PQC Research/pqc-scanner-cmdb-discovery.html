1. What a CMDB Is

A CMDB (Configuration Management Database) is a company's master list of every technology asset it owns. Each item is called a CI (Configuration Item) - a server, a piece of software, a certificate, a network service, an application, and so on.

The CMDB also stores relationships between CIs: this software is installed on that server, this application depends on that database, this certificate was issued by that CA. These relationships are what make a CMDB more than a spreadsheet.

2. Approach A: CMDB-Based Discovery

If a CMDB exists and is reasonably current, the PQC scanner can use it as the primary inventory source. The integration has four layers.

3. Approach A.1: How the Tool Connects

A CMDB is just software, and like all modern software it exposes ways for other programs to talk to it.

ChannelExamplesMechanism
REST APIServiceNow Table API, NetBox, Device42, BMC HelixHTTPS request/JSON, OAuth2 or API key
Direct database readCustom CMDBs on Postgres, MySQL, MSSQLRead-only SQL account
File exportLegacy CMDBs, Excel-driven shopsScheduled SFTP/S3 CSV or JSON dumps
Event streamServiceNow webhooks, Kafka change-data-captureSubscribe to CI add/update/delete events

In all cases the tool uses a service account - a dedicated username and password or API key with read-only access, so normal user permissions do not matter and nothing breaks if the account is misused.

4. Approach A.2: How the Tool Reads the CMDB

The CMDB does not return one big list. It returns separate classes (tables) that the tool has to join together.

Relevant CMDB classes

  • Hardware/server CIs - hostname, IP, OS, environment (prod/dev), owner
  • Software CIs and software instances - what is installed where, with version (e.g., OpenSSL 1.0.2 on server X)
  • Certificate CIs - the cert itself: issuer, algorithm, key size, expiry, where it is deployed
  • Key, HSM, KMS CIs - where the private keys live
  • Application and service CIs - the business-level app that ties everything together
  • Relationship records - the links between all of the above

Reading pattern

  1. Schema discovery - ask the CMDB what classes and fields it has, cache the result.
  2. Federated query - pull a CI, pull its relationships, pull the related CIs, repeat.
  3. Initial full pull - batch, paginated.
  4. Incremental sync - after the first run, fetch only CIs that changed since the last pull, using a timestamp like last_updated > X.

5. Approach A.3: How It Filters for PQC

The CMDB may hold over 100,000 CIs. The tool picks only the ones relevant to cryptography using four filter layers.

Layer 1 - Class filter

Keep only CIs that can host or use cryptography: servers, load balancers, VPN gateways, web apps, APIs, databases, certs, keys, HSMs, code repos, build pipelines.

Layer 2 - Attribute and keyword filter

Within those, keep CIs whose description, tags, or named fields contain words like tls, ssl, cert, key, hsm, kms, ca, vpn, ssh, pgp, signature, jwt. This cuts noise from servers that do not actually touch crypto.

Layer 3 - Relationship traversal

A CA server by itself is not the problem - it is a problem because 200 certificates depend on it. From a known crypto-relevant CI, the tool walks the relationship graph outward:

  • Show every CI that depends on this OpenSSL instance
  • Show every CI that uses a certificate issued by this CA

This is what turns a list into a useful inventory.

Layer 4 - Priority filter

PQC migration takes years. Sort by environment = production and business_criticality = high/critical first. Dev, test, and internal-only systems come later.

PQC-specific narrowing

Within crypto, only asymmetric primitives are quantum-vulnerable - RSA, ECDSA, ECDH, DH, DSA. Symmetric primitives (AES, SHA-2/3) at sufficient key and hash sizes are considered safe. So the inventory focuses on:

  • Public-key algorithms
  • PKI and certificate chains
  • Key exchange
  • Digital signatures
  • Code signing
  • Protocols that use them: TLS, SSH, S/MIME, PGP, JWT, IPsec

6. Approach A.4: Mapping Component Relationships

This is the part that answers the question that matters most: if I have to change this crypto thing, what else breaks?

The dependency graph

The CMDB already stores the relationships as typed edges. The PQC tool's job is to build a dependency graph where nodes are CIs and edges are relationships, then run traversals on it.

Business Service (e.g., "Customer Portal")
  ↓ uses
Application (e.g., "Web App v3.2")
  ↓ has
Software Instance (e.g., "OpenSSL 1.0.2")
  ↓ runs on
Host (e.g., "web-prod-42")
  ↓ exposes
Network Endpoint (TLS 1.2, :443)
  ↓ presents
Certificate (RSA-2048, expires 2027)
  ↓ signed by
Private Key

The four traversals that drive PQC planning

TraversalQuestionDirection
Impact analysisIf I disable RSA on this CA, what services break?Walk Issued by / Used by backwards
Blast radiusIf I upgrade OpenSSL on this host, which apps are affected?Walk Runs on / Installed on / Used by forward
Crypto lineageWhich business services depend on a quantum-vulnerable algorithm?Algorithm node up to business service
Cert expiry vs migration windowWhich certs will outlive the migration plan?Compare cert expiry to migration roadmap

Graph storage pattern

Most teams do not run traversals inside the CMDB itself - CMDB UIs are not built for it. The common pattern is to extract a PQC-relevant slice of the CMDB (CIs, relationships, crypto findings) into a graph database. Neo4j is the typical choice. The PQC tool runs traversals against that graph, and the visualization is a graph view, not a table.

Write-back loop

Findings are pushed back into the CMDB as new CIs or as fields on existing CIs, linked via relationships. This is what closes the loop - the company's existing change-management, ticketing, and ownership workflows (which already live in the CMDB) automatically pick up the PQC remediation work. Nobody has to maintain a parallel system.

7. Approach B: No-CMDB Discovery (From Scratch)

Most companies, even large ones, do not have a usable CMDB, or have one that is so stale it cannot be trusted. "No CMDB" is the more common starting point. Without a CMDB the tool has to find the assets instead of being told about them. It does this by looking at the technology from many different angles and then merging the results into one picture. No single method gives the full answer; you need several running in parallel and then reconciled.

There are five discovery sources.

8. Approach B.1: Network-Level Discovery

Scan the company's IP ranges to find live hosts, open ports, and running services. For each service that uses TLS or SSH, do a handshake - actually connect and read the certificate and the cryptographic parameters the server offers.

Per endpoint this gives:

  • The certificate (issuer, algorithm, key size, expiry, serial)
  • The TLS version supported
  • The key exchange and signature algorithms offered
  • The cipher suites

Tools: Nmap and Masscan for port scanning; testssl.sh, sslscan, and CryptCheck for the crypto handshake; or commercial scanners (Qualys, Tenable, Rapid7) that do the same.

This is the fastest way to get a first inventory of public-facing and reachable crypto, but it only sees what is listening on a port.

9. Approach B.2: Certificate Authority Queries

Every cert a company uses was issued by something. If you can ask the issuer, you have a complete list - not just the certs that happen to be on a live port today.

Query the actual CA systems:

  • Active Directory Certificate Services (AD CS) - pull every cert it has issued, active, or revoked. This is usually where 70-80% of internal Windows certs live.
  • AWS Private CA, Azure Key Vault, Google CAS - same idea, via the cloud provider's API.
  • HashiCorp Vault PKI engine, Smallstep, EJBCA - for non-Microsoft PKI.
  • Public Certificate Transparency (CT) logs - passive, public databases of every cert issued by a public CA. Useful for catching certs you did not know existed (subsidiaries, shadow IT, dev teams using Let's Encrypt directly).

10. Approach B.3: Identity, Keys, and HSM Systems

Cryptographic keys are the most sensitive inventory item. Pull them from where they are stored:

  • HSM inventories - AWS CloudHSM, Azure Dedicated HSM, Thales Luna, Entrust nShield. Each has an admin API that lists keys, algorithms, and key sizes.
  • KMS systems - AWS KMS, Azure Key Vault, GCP KMS, HashiCorp Vault. List keys and their declared algorithms.
  • SSH infrastructure - SSH CA servers (if used), known_hosts files from config management, jump host inventories.
  • Code-signing systems - HSM-backed signing services, Sigstore, in-toto. These list which keys are allowed to sign what.

11. Approach B.4: Software and Code Inventory

The actual crypto is implemented in libraries. You need to know which libraries, which versions, and where:

  • Endpoint agents - install a lightweight agent (osquery, Wazuh, Tanium, custom) on servers and workstations to enumerate installed crypto libraries: OpenSSL, BoringSSL, JCE/BouncyCastle, .NET CryptoAPI, Go crypto/tls, mbedTLS, wolfSSL, OpenSSH, LibreSSL, NSS, GnuTLS, libsodium.
  • Software Bill of Materials (SBOM) - pull SBOMs from the build pipeline (CycloneDX, SPDX format). This is the most accurate source because it is generated at build time.
  • Software Composition Analysis (SCA) - Snyk, Mend, Black Duck, Dependabot, Trivy. Scans source repos and container images for crypto library versions and known CVEs.
  • Static source analysis - grep, Semgrep, or custom rules to find direct crypto API usage, hardcoded keys, and custom crypto implementations. Important for catching hand-rolled crypto that scanners miss.
  • Container image scanning - Trivy, Grype, Clair. Most production crypto now runs in containers; the image SBOM is the inventory.

12. Approach B.5: Cloud and SaaS Configuration

Modern infrastructure lives in cloud accounts and SaaS platforms. Their APIs can tell you what is configured:

  • Cloud asset inventory - AWS Config, Azure Resource Graph, GCP Cloud Asset Inventory. Lists every resource (load balancers with TLS, API Gateway, CloudFront) and its TLS settings.
  • Cloud IAM - which service accounts have key material, and where keys are stored.
  • SaaS configs - Okta, Auth0, Microsoft Entra ID, Workday, Salesforce. These have admin APIs that list signing keys, certs, and algorithm settings.
  • Email security - DNS records (SPF, DKIM, DMARC, MTA-STS) tell you which keys sign your email.

13. The Phased Rollout Sequence

The order matters because each phase gives you information that makes the next phase more targeted.

Phase 1 - Public-facing crypto (low-hanging fruit)

External TLS scan of your known IP ranges, plus CT log monitoring, plus public DNS and repo scan. You get a first inventory in days, focused on the things a quantum attacker would hit first.

Phase 2 - Certificate authorities (the backbone)

Pull from AD CS, cloud CAs, Vault, EJBCA. This is the single biggest source of internal cert truth and usually reveals certs no one knew existed.

Phase 3 - Internal network scanning

Scan internal IP ranges for TLS and SSH endpoints. Cross-reference with Phase 2 to find:

  • CAs that exist but are not actively issuing (good to know)
  • Certs found in the wild that are not in any CA (orphans - a security red flag)
  • Endpoints with no cert at all (probably still using defaults)

Phase 4 - Key stores and HSMs

Enumerate KMS, HSM, Vault. This gives you the asymmetric private keys - the actual PQC migration target.

Phase 5 - Software and SBOM

Deploy endpoint agents or pull SBOMs from build pipelines. This gives you the library versions that need upgrading.

Phase 6 - Code and cloud

Static analysis for hand-rolled crypto, cloud config audits, SaaS key inventories.

14. Building the Picture: Reconciliation and Inference

Without a CMDB, you do not have a relationship graph - you have five overlapping lists. The tool's job is to reconcile them into one inventory.

Deduplication

The same certificate can show up in three sources: the network scan saw it on port 443, AD CS issued it, and Vault has the key. The tool needs to recognize it is the same cert (by serial number, fingerprint, or subject+SAN match) and merge the records.

Relationship inference

This is where you build the graph that the CMDB would have given you. You infer relationships from evidence:

EvidenceInferred relationship
Same hostname in network scan and software inventorySoftware X runs on host Y
Cert CN/SAN matches a service hostnameCert is used by service Y
Issuer DN matches a CA's CNCert was issued by CA Z
Library X found on host Y via SBOM, and host Y is in the software inventory of service ZService Z depends on library X
Same owner tag, same project label, same cost centerOwned by team T
IP in same subnet, same VLAN tag, same cloud VPCNetwork locality

These inferences are probabilistic, not certain. The tool should assign a confidence score to each inferred relationship so a human can review the weak ones.

Continuous refresh

Unlike a CMDB pull (where the CMDB is the source of truth and you just sync), from-scratch discovery has to run on a schedule: weekly network scans, daily CA pulls, continuous SBOM ingestion, real-time CT log monitoring. Assets change, certs get reissued, libraries get patched.

15. Practical Realities

  • Pick the lowest-friction sources first. CT logs, AD CS queries, and SBOM feeds are often available in days. Endpoint agents and full network scans take months of deployment.
  • You will find things nobody owns. A cert on a server no one remembers. A private key in a developer's home directory. A VPN gateway in a subsidiary's network. This is normal and one of the main reasons to do the inventory.
  • Build a real asset database as you go. Even if you started with no CMDB, by month 6 you should have a PQC inventory database that is, in effect, a crypto-focused CMDB. This becomes the foundation for everything else - migration tracking, remediation tickets, executive reporting.
  • Manual surveys still help. For areas scanners cannot reach (third-party vendors, embedded devices, legacy mainframes), a questionnaire to system owners is the only way. The tool should support importing those answers.

16. CMDB vs No-CMDB: Trade-offs

DimensionCMDB-basedNo-CMDB (from scratch)
Time to first inventoryDays (if CMDB is good)Weeks to months
Relationship graphAuthoritative, already builtInferred from evidence, with confidence scores
Ownership dataUsually present in CMDBOften missing; requires manual surveys
Coverage of cryptoOnly what is registered in the CMDBAll crypto in the organization, including shadow IT
Continuous syncIncremental pull, simpleScheduled scans, more moving parts
Data quality riskTrusts CMDB; garbage in, garbage outEach source has its own quality issues
Best fitMature enterprise with good CMDB hygieneMost companies, especially at the start

Realistic recommendation: do not treat these as alternatives. The CMDB (if it exists) is the inventory backbone; the from-scratch sources fill the gaps. A mature PQC scanner uses both in parallel - CMDB for the graph and ownership, network/CA/SBOM sources for the actual crypto facts the CMDB does not record.

17. Glossary

CI: Configuration Item - a single tracked asset in a CMDB.

CMDB: Configuration Management Database.

CT logs: Certificate Transparency logs - public, append-only records of certificates issued by public CAs.

Federated query: A query that pulls related records across multiple CMDB classes in one operation.

HSM: Hardware Security Module - a physical device that stores and uses cryptographic keys.

HSM inventory: The list of keys, algorithms, and key sizes held by an HSM.

Incremental sync: Fetching only records that changed since the last pull.

KEM: Key Encapsulation Mechanism - a post-quantum public-key primitive for key exchange.

KMS: Key Management Service - a software system for storing and using cryptographic keys.

PKI: Public Key Infrastructure.

PQC: Post-Quantum Cryptography.

SBOM: Software Bill of Materials - a machine-readable list of components in a software build.

SCA: Software Composition Analysis - tools that detect libraries and known vulnerabilities in source code or container images.

Service account: A non-human login used by software, typically with scoped permissions.

Schema discovery: The act of asking a system what classes and fields it has, before querying data.

Traversals: Queries that walk a graph along its edges to find related nodes.
