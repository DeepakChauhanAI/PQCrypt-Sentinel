Executable Decompilation & Recovery Flow
Extracting Python source files from the packaged PyInstaller Windows Binary
Core Summary The executable PQC_Network_Scanner_2.0.exe is a packaged Python application built using PyInstaller. Through a multi-stage static analysis, we successfully unpacked the executable archive and retrieved the complete, original source code.
During standard packaging, PyInstaller compiles all local Python modules into compiled bytecode (.pyc files) and bundles them inside a compressed CArchive, appended to a bootloader. Standard decompilers fail on the raw binary, requiring a structured reverse engineering pipeline.
Step 1: PyInstaller Archive Extraction
We utilized pyinstxtractor.py to parse the executable's structural layout. This tool reads the executable, locates the signature cookie at the end of the file, extracts the embedded CArchive directory structure, and unpacks the files.
COMMAND EXECUTED:
# Execute the PyInstaller Extractor to unpack the binary python pyinstxtractor.py PQC_Network_Scanner_2.0.exe
The extraction process identified 1,016 files bundled in the CArchive and 218 files in the PYZ archive. The main entry point was identified as:
PQC_Network_Scannerv2.pyc (Python 3.11 Bytecode, Size: 75,931 bytes).
Step 2: Bytecode Decompilation & Compiler Building
Because the executable was packaged using Python 3.11, traditional Python-based decompilers like uncompyle6 and decompyle3 failed due to significant bytecode structural updates in Python 3.11+.

To resolve this, we cloned the repository for Decompyle++ (pycdc), configured it with CMake, and compiled it locally using the Microsoft Visual C++ Compiler (MSVC) to generate a native Windows decompiler executable.
COMPILER COMMANDS EXECUTED:
# Clone and compile the pycdc C++ bytecode analyzer git clone https://github.com/zrax/pycdc.git cd pycdc cmake . cmake --build . --config Release
We ran the compiled pycdc.exe on the extracted PQC_Network_Scannerv2.pyc bytecode:

Core Code Architecture
Function-by-function analysis of PQC_Network_Scannerv2.py
The entire scanning application is self-contained within one Python module. It is divided into three functional domains: network connection/evasion, cryptographic parsing/risk assessment, and Tkinter GUI management.
Domain A: Connection Pacing & Evasion
class RateLimiter
Constructor: max_rate (float), jitter (float) | Thread-Safe
Limits the rate of connection requests to avoid overwhelming target endpoints and evade basic Intrusion Detection System (IDS) alerts.

Wait Loop (wait method): Calculates the time elapsed since the last request. If it is less than the target interval, it sleeps for the difference. Integrates a random uniform jitter value to break predictable request timing intervals.
Rate Pacing
Timing Randomization
apply_throttling(mode_config, scan_count, error_count)
Helper Function | Throttling Engine
Suspends the thread execution based on the chosen throttling profile:
  • Normal: 0.0s delay (full-speed scanning).
  • Slow: 2.0s to 5.0s randomized delay + 0.5s jitter.
  • Stealth: 5.0s to 15.0s randomized delay + 2.0s jitter (maximum evasion).
  • Random: 1.0s to 10.0s randomized delay + 3.0s jitter.
  • Adaptive: Smart timing adjustment. Starts 1.5x slower for the first 10 scans. If error rates exceed 20%, it doubles the base delay.

Domain B: Cryptographic Inspection
fetch_server_certificate(host, port, timeout)
Function | Network Handshake
Initiates a raw socket connection, wraps it in a default SSL context (ignoring certificate verification for speed/reach), and executes the TLS handshake.

Certificate Extraction: Retrieves the leaf certificate via sock.getpeercert(binary_form=True) and attempts to extract the full chain using get_unverified_chain() or getpeercertchain(). Returns the chain, cipher suite, and TLS version.

detect_pqc_hybrid_features(cert, cipher_info)
Function | Signature & Key Matching
Analyzes the extracted X.509 certificate for Post-Quantum Cryptography indicators:
  1. Checks the signature algorithm OID against a dictionary of standardized or draft PQC algorithms (Dilithium, Falcon, SPHINCS+).
  2. Performs pattern matching on the signature name string for PQC keywords.
  3. Inspects the public key class name for PQC keywords (e.g., kyber, mlkem).
  4. Checks certificate extensions for OIDs indicating PQC or composite key usage.
  5. Inspects the Subject Alternative Name (SAN) and Subject/Issuer DN fields for test markers.
  6. Checks the negotiated TLS cipher suite for hybrid key exchange patterns (e.g., X25519-Kyber768).

analyze_certificate(host, port, throttle_config, ...)
Function | Orchestrator & Risk Scorer
Combines DNS resolution, device profiling, network throttling, handshake, chain analysis, and hazard scoring:
  • RSA Keys: Flagged as High severity if key size < 4096 bits, or Medium if ≥ 4096 bits. Urgency is downgraded to Medium (for hybrid) or Low (for pure) if PQC features are present.
  • Elliptic Curve (EC) Keys: Flagged as High severity since ECDSA/ECDH is vulnerable to Shor's algorithm.
  • Deprecated TLS: Elevates risk to High if TLS version ≤ 1.1 or if TLS 1.2 is negotiated without Perfect Forward Secrecy.
