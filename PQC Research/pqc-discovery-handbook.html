Part 1 — Enterprise Infrastructure Fundamentals
Layered modeling of cryptographic dependencies in corporate assets
1.1 Why this part exists

Before we can discover cryptography, we have to know what an enterprise is. Cryptography is not a separate system. It is a property that lives inside almost every system. A scanner that does not understand the surrounding architecture will collect evidence but cannot reason about it.

The goal of Part 1 is to give every reader a shared mental model of what categories of assets exist in a typical enterprise, how those assets are layered, and where cryptography actually appears in each layer.

1.2 Asset taxonomy

A modern enterprise contains at least the following categories of assets. This list is not exhaustive; it is the baseline that any PQC discovery programme must cover.

Compute
  • Endpoints (laptops, desktops, mobile devices)
  • Servers (physical and virtual) & VMs
  • Containers & Kubernetes clusters
  • Serverless platforms & Mainframes
  • Edge / IoT devices
Network
  • Routers & Switches (L2/L3)
  • Firewalls (NGFW, WAF) & Load balancers (L4/L7)
  • VPN concentrators & Proxies/Reverse proxies
  • DNS servers & DHCP / IPAM
  • Network taps, packet brokers & SD-WAN edges
Identity and access
  • Active Directory / LDAP directories
  • Identity providers (SAML, OIDC, OAuth)
  • Certificate Authorities (internal, public, private)
  • HSMs & key management appliances
  • Smart card systems (PIV, CAC) & PAM vaults
  • SSO / federation gateways
Storage and data
  • Databases (relational, NoSQL, graph)
  • Object storage (S3-compatible)
  • File shares (SMB, NFS)
  • Backup & Archive systems
Applications
  • Web applications and APIs
  • Internal services (microservices, ESBs)
  • Email (Exchange, Postfix, Gmail)
  • Code signing & CI/CD platforms
  • Package registries & Document signing
Hardware & Roots of Trust

Cloud and SaaS

Operational

  • Config management (Ansible, Puppet, Intune)
1.3 The layer model

Every asset can be decomposed into the same four layers, top down. Cryptography appears at every layer, and the scanner must look at all four:

L1
Users
The human or service identity that initiates a cryptographic operation.
Discovery targets: Authentication methods (smart card, FIDO2, PKINIT), identity providers, certificate-based user identities (S/MIME, signing), and session establishment (TLS, SSH, VPN).
L2
Applications
The software components executing application logic. This is the most diverse layer.
Discovery targets: TLS endpoints and certificate bindings, application certificate stores (Java keystores, IIS, Nginx), API auth (JWT, mTLS, OAuth assertions), code signing pipelines, and internal crypto libraries (OpenSSL, NSS, JCE, CNG, .NET).
L3
Operating Systems / Runtimes
The OS exposes the cryptographic stack to applications.
Discovery targets: OS certificate stores (Windows Cert:\, Linux NSS, macOS Keychain), SSH config, default TLS libraries (Schannel, system OpenSSL), kernel module signing, OS update verification, disk encryption (BitLocker, LUKS), smart card middleware (PKCS#11), and TPM/PKI integrations.
L4
Hardware / Firmware
The underlying physical root of trust.
Discovery targets: TPM presence and spec capabilities, UEFI/Secure Boot variables (PK, KEK, db, dbx), HSM key inventories, smart card reader details, hardware tokens, and BIOS/firmware update signatures.
1.4 Why the layer model matters for the scanner

The same algorithm (RSA-2048) can appear at every layer. The scanner must collect evidence at the right layer or it will produce the wrong risk picture. A finding of "RSA-2048 found" without context is useless.

Layer Example finding Risk if missed
Hardware TPM 2.0 supports RSA-2048 only Underestimates the cost of TPM-backed attestation migration.
Firmware Secure Boot db contains RSA-2048 CA Misses the need to re-issue firmware trust anchors.
OS OpenSSH supports ssh-rsa Overstates migration scope if ssh-rsa is only listed, not preferred.
Application Java app uses JCE provider SunRsaSign Misses application-specific crypto libraries.
1.5 Access tiers

Every discovery command in this handbook maps to an access tier. The scanner must request the minimum tier required for each task, escalating only when a lower tier cannot produce the required evidence.

Tier Description Typical Use
Tier 0 Unauthenticated (Public network access) Port scans, banner grabs, active TLS probes, network inspection
Tier 1 Authenticated (Read-only user) SSH audit, standard certificate store read, local config parse
Tier 2 Privileged (Local Admin / Root) WMI query, iptables inspection, TPM2 direct access, UEFI variable read, AD CS audit
Tier 3 Hardware / Out-of-band HSM key inventory, physical tokens, smart card certificate extraction, HSM PED operations
Tier 4 Cloud / Control plane Cloud provider APIs (AWS Config, Azure Resource Graph, GCP Asset Inventory)

Part 3 — Discovery Playbooks (3.1 - 3.7)
Network, OS, and Identity Cryptographic Playbooks
3.1 TLS Endpoints
Layer 2: Application / Transport
Why it matters TLS protects almost every enterprise protocol: HTTPS, LDAPS, SMTPS, IMAPS, database connections, and microservices. It is the single largest surface for asymmetric cryptography in most enterprises. A PQC readiness assessment that misses TLS is incomplete.
Discovery Methods

Windows (IIS / Schannel): Enumerate active bindings using Get-WebBinding or netsh http show sslcert. Resolve bindings to thumbprints and verify details in the Cert:\LocalMachine\My store.
Linux (Web Servers): Parse vhost configs for ssl_certificate locations (Nginx, Apache, HAProxy). Retrieve cert details using openssl x509 -text -noout -in cert.pem.
Active Probes (Any Platform): Probing with tools such as testssl.sh (run in JSON mode for ingestion) or `nmap --script ssl-enum-ciphers,ssl-cert`.

ACTIVE PROBE COMMANDS:
# Force specific protocol check & retrieve negotiation details openssl s_client -connect host:443 -tls1_2 curl -v --tls-max 1.2 https://host
Access & Evidence
Access Tier Evidence Captured What it cannot determine
Tier 0 (Network Access)
Tier 2 (Local cert stores)		 Protocol versions, Cipher suites, Key exchange groups (X25519, P-256), Sign algs, Certificate chain thumbprints. The origin server if terminated by a reverse proxy. Whether the negotiated ciphers are actively used by real clients.

Inventory fields: endpoint, protocol_versions[], kex_groups[], signature_algs[], cipher_suites[], server_cert_thumbprint, server_cert_sig_alg, pqc_capable (bool), pqc_groups[]

PQC Relevance TLS is the primary target of NIST standards. ML-KEM (Kyber) replaces ECDHE for key exchange; ML-DSA (Dilithium) replaces RSA/ECDSA for signatures. Hybrid deployment (e.g., X25519MLKEM768) is current best practice.
3.2 SSH
Layer 3: OS / Application
Why it matters SSH cannot establish trust or secure key exchange without asymmetric cryptography. The SSH surface spans hundreds to thousands of hosts. Algorithm choice on each host determines the migration scope.
Discovery Methods

Active Handshake Dump: Run a verbose connection to see negotiation features.

ssh -v user@server 2>&1 | grep -E "kex|cipher|host key"

Server Configuration: Run ssh -Q kex or ssh -Q key to query supported algorithms, and parse local configuration from /etc/ssh/sshd_config (Linux) or %ProgramData%\ssh\sshd_config (Windows).

ANSIBLE SCALE PLAYBOOK (ssh_audit.yml):
- name: Audit SSH Configuration hosts: all tasks: - name: Get sshd config ansible.builtin.slurp: src: /etc/ssh/sshd_config register: sshd_cfg
Access & Evidence
Access Tier Evidence Captured What it cannot determine
Tier 1 (Read Config / Query SSH) Host key algs (ssh-rsa, ssh-ed25519), KEX algs, Ciphers (aes256-gcm), MACs, config lines. Whether configured key algorithms are actively utilized by real client endpoints. Trust chains of host keys.

Inventory fields: host, sshd_version, host_key_algs[], kex_algs[], ciphers[], macs[], pubkey_accepted_algs[], pqc_capable (bool)

PQC Relevance OpenSSH supports experimental hybrid KEX (sntrup761x25519, mlkem768x25519) and signature algs (dilithium, falcon). Migration requires updating both server configs and client software.
3.3 VPN
Layer 1: Network
Why it matters VPN concentrators terminate enterprise-wide encrypted tunnels. Their IKE, IPsec, and TLS configuration determines the cryptographic posture of every remote connection.
Discovery Methods

IPsec / IKEv2: Read concentrator config files (Cisco, Fortinet, Palo Alto). Inspect IKE proposals (DH group, PRF, lifetime) and Phase 2 PFS groups.
SSL-VPN: Read local configs (e.g. openvpn.conf for --tls-cipher directive) and run active probes on HTTPS VPN portals using openssl s_client.
Active probes: Enumerate endpoints using ike-scan or nmap --script ike-version.

Access & Evidence
Access Tier Evidence Captured What it cannot determine
Tier 2 (Network Admin Config Read) IKE proposals, Phase 2 PFS groups, Auth methods (RSA cert, PSK), VPN tunnel configurations. Whether legacy proposals are reachable in production or firewalled. Proprietary vendor algorithms.

Inventory fields: concentrator, protocol, ike_proposals[], pfs_group[], auth_method, server_cert_thumbprint, pqc_capable (bool)

PQC Relevance IPsec/IKEv2 standards are adopting hybrid PQC key exchanges (ML-KEM with DH groups). Vendor support varies. Migration impacts appliance firmware, client software, and PKI certs.
3.4 Certificate Stores
Layer 3: OS / Application
Why it matters The OS certificate store is the trust anchor for TLS, S/MIME, code signing, and client authentication. A trusted certificate defines what algorithms the OS will trust to validate downstream components.
Discovery Methods

Windows: Enumerate certificates via PowerShell (e.g. Get-ChildItem Cert:\LocalMachine\Root).
Linux (NSS Shared DB): Enumerate using NSS tools: certutil -L -d sql:/home/user/.pki/nssdb.
macOS: Extract certificates via Security utility: security find-certificate -a -p /Library/Keychains/System.keychain.
Certificate Inspection: Parse files using openssl x509 -in cert.pem -text -noout to check OIDs.

RESOLVING MAPPING GRAPH:
Component Config (IIS Thumbprint / Nginx File Path) ↓ Search Cert Store for matching reference ↓ Extract Algorithm OID (e.g., RSA-2048 vs ML-DSA)
Access & Evidence
Access Tier Evidence Captured What it cannot determine
Tier 1 (Read Local Stores) Store location, Subject/Issuer, Thumbprint, PubKey alg, PubKey size, keyUsage, self_signed status. Which applications are actively calling the certificate. Whether the certificate is actually active or orphaned.

Inventory fields: thumbprint, subject, issuer, not_before, not_after, sig_alg, pub_key_alg, pub_key_size, key_usage[], eku[], self_signed, store_location, pqc_capable (bool)

3.5 Active Directory and PKI
Layer 3: OS / Identity
Why it matters Active Directory and AD Certificate Services (AD CS) manage the majority of internal credentials, trust anchors, Kerberos tickets, and domain TLS configurations.
Discovery Methods

CA and Template Enumerate: Enumerate Active Directory PKI objects using certutil -config - -ping, certutil -CATemplate, and AD commands like Get-ADObject -Filter {objectClass -eq "certificationAuthority"}.
Domain Controller Certs: Inspect certificates local to Domain Controllers used for Kerberos PKINIT and LDAPS.

AUDIT AD CS VIA TOOLING:
# Analyze GPO applied crypto policies gpresult /h gpo.html Get-GPResultantSetOfPolicy -ReportType HTML -Path gpo.html
Access & Evidence
Access Tier Evidence Captured What it cannot determine
Tier 2 (Domain/Enterprise Admin) CA lists, CA key sizes & signature algs, template configurations, Kerberos encryption types, GPO cipher ordering. Offline root CAs and their hardware key handling. Certificates issued to endpoints currently offline.

Inventory fields: ca_name, ca_sig_alg, ca_key_size, templates[], template_min_key_size, dc_cert_sig_alg, kerberos_enc_types[], gpo_cipher_order[], pqc_capable (bool)

PQC Relevance KDC keys and domain controllers are highly valuable targets. Microsoft and industry working groups have published explicit PQC guidance for Active Directory PKINIT transitions.
3.6 TPM
Layer 4: Hardware / Firmware
Why it matters TPMs are secure hardware chips that store keys non-exportably for disk encryption, device identity, and Secure Boot measurements. TPM limitations directly impact attestation systems during PQC migration.
Discovery Methods

Windows: Query local TPM state via Get-Tpm cmdlet.
Linux: Enumerate properties and algorithms via tpm2-tools commands:

sudo tpm2_getcap properties-fixed sudo tpm2_getcap algorithms sudo tpm2_listpersistent
Access & Evidence
Access Tier Evidence Captured What it cannot determine
Tier 2 (Local Admin / Root) TPM present, TPM version (e.g. 2.0), Manufacturer ID, supported algorithms, persistent objects count. BitLocker recovery key state (requires Active Directory query). Real-time application utilization.

Inventory fields: tpm_present, tpm_version, manufacturer, spec_version, firmware_version, algs[], pcr_banks[], persistent_object_count, attestation_capable, pqc_capable (bool)

PQC Relevance Most production TPMs (TCG 1.38) support RSA and ECC only. Next-gen TPMs (TCG 1.59+) introduce ML-KEM/ML-DSA. attestation chains will require hardware replacement or hybrid firmware updates.
3.7 HSM
Layer 4: Hardware / Firmware
Why it matters HSMs are high-assurance appliances hosting the keys for root CAs, transaction processors, EV code signing, and TLS offloading. Replacing or updating HSM configurations requires long lead times.
Discovery Methods

PKCS#11 Interfaces: Scan application folders for PKCS#11 references, then run pkcs11-tool --module /usr/lib/opensc-pkcs11.so --list-slots.
Vendor CLI Tools: Call proprietary management modules (e.g. Thales LunaCM, Entrust keytool).
Cloud HSM KMS: Audit KMS protected resources via cloud management APIs.

Access & Evidence
Access Tier Evidence Captured What it cannot determine
Tier 3 (Hardware partitions / PIN)
Tier 4 (Cloud IAM Read)		 HSM Vendor (Thales, Entrust), FIPS validation level, key labels, key sizes/algorithms, key usage flags. The actual private key material (secured by design). Backup key escrow physical locations.

Inventory fields: hsm_id, vendor, model, fips_level, key_label, key_alg, key_size, key_usage[], state, rotation_policy, pqc_capable (bool)

Part 3 — Discovery Playbooks (3.8 - 3.15)
Hardware, Platform, and Application Cryptographic Playbooks
3.8 Smart Cards
Layer 4: Hardware / Token
Why it matters Smart cards store non-exportable identities for PIV/CAC user auth and digital signing. Migrating smart card infrastructure is historically slow and expensive.
Discovery Methods

Windows: Enumerate readers with Get-PnpDevice | Where-Object { $_.Class -eq "SmartCardReader" }, and extract card data with certutil -scinfo.
Linux: Detect readers using opensc-tool -l, list slots with pkcs11-tool --list-objects, and pull certificates for parsing.

PYTHON READER CHECK (pyscard):
from smartcard.System import readers print("Connected Readers:", readers())
Access & Evidence
Access Tier Evidence Captured What it cannot determine
Tier 1 (Endpoint Agent Read) Reader vendor/model, Card ATR (Answer-to-Reset), Certificate subject, issuing CA, public key size and alg. PIN policies (length, retries). Whether the card is actively used for transaction authentication.

Inventory fields: reader_vendor, reader_model, card_atr, card_type, cert_subject, cert_issuer, pub_key_alg, pub_key_size, sig_alg, pqc_capable (bool)

3.9 Secure Boot (UEFI)
Layer 4: Firmware / Boot
Why it matters UEFI Secure Boot guarantees system bootloader and kernel integrity. A compromised chain or un-migrated Secure Boot db/dbx presents major root-of-trust risks.
Discovery Methods

Windows: Query basic status via Confirm-SecureBootUEFI. Read variable structures via Get-SecureBootUEFI.
Linux: Check state with mokutil --sb-state and read variables via efi-readvar.

POWERSHELL DATABASE DUMP:
Get-SecureBootUEFI -Name PK Get-SecureBootUEFI -Name KEK Get-SecureBootUEFI -Name db Get-SecureBootUEFI -Name dbx
Access & Evidence
Access Tier Evidence Captured What it cannot determine
Tier 2 (Local Admin / Root) Secure Boot status, PK/KEK subjects and algs, signature database (db) algs, dbx size/entry counts. Measured Boot state (requires PCR verification). firmware upgrade paths for PQC compat.

Inventory fields: secure_boot_enabled, uuefi_firmware_type, pk_alg, kek_algs[], db_algs[], dbx_count, dbx_size, firmware_version, pqc_capable (bool)

3.10 Disk Encryption
Layer 3: OS / Hardware
Why it matters FDE (BitLocker, LUKS, FileVault) protects stationary data. While symmetric AES-256 is quantum-safe, the key wrapping and TPM seals are asymmetric and require review.
Discovery Methods

Windows: Query using cmdlets manage-bde -status or Get-BitLockerVolume.
Linux: Dump header variables using cryptsetup status /dev/sda2 and cryptsetup luksDump /dev/sda2.
macOS: Query using fdesetup status.

Access & Evidence
Access Tier Evidence Captured PQC Relevance
Tier 2 (Local Admin / Root) Encryption method (XtsAes256), Key size, protector (TPM/password), KDF (SHA-256, Argon2id). Low for symmetric bulk cipher. Medium for asymmetric key protectors (TPM seals).
3.11 Code Signing
Layer 2: Application / Software Supply Chain
Why it matters Code signing verifies software authenticity. Compromised, weak, or expiring code-signing keys undermine trust in software delivery pipelines.
Discovery Methods

Windows: Run Get-AuthenticodeSignature "C:\Path\To\File.exe". Enumerate signed drivers via Get-WindowsDriver -Online.
macOS: Extract signatures with codesign -dv --verbose=4 /Applications/Sample.app.
Packages: Audit system public key registers (e.g. apt-key list or rpm databases).

Access & Evidence
Access Tier Evidence Captured What it cannot determine
Tier 1 (User read access) File paths, Signer subjects/issuers, Signature algorithm, Public key algorithm, RFC 3161 timestamps. The location of HSM or software keys used to generate the signatures during CI/CD.

Inventory fields: file_path, signer_subject, signer_issuer, sig_alg, pub_key_alg, timestamp_authority, status, pqc_capable (bool)

3.12 Kubernetes
Layer 2: Orchestration / Application
Why it matters Kubernetes runs service-to-service communication. Weak certs in the control plane, etcd configurations, or service mesh parameters degrade cluster posture.
Discovery Methods

Cluster Control Plane: Query cluster configurations and parse API server certificate anchors.

kubectl version kubectl config view --raw -o jsonpath='{.clusters[0].cluster.certificate-authority-data}' | base64 -d | openssl x509 -text -noout

Mesh and Ingress: Audit service mesh SDS endpoints using istioctl proxy-config secret and check Ingress TLS blocks.

Access & Evidence
Access Tier Evidence Captured What it cannot determine
Tier 2 (Kubectl read permissions) API server cert sigs, etcd encryption configurations (AES-CBC/GCM), Service mesh identities, Ingress settings. Underlying container OS library features if running unprivileged workloads.
3.13 Cloud KMS
Layer 4: Cloud Control Plane
Why it matters Cloud KMS handles central key infrastructure in public clouds. Reviewing key types, usage parameters, and policies is critical for cloud workloads.
Discovery Methods

AWS KMS: List and describe keys via CLI:

aws kms list-keys --region us-east-1 aws kms describe-key --key-id <arn> aws kms get-key-rotation-status --key-id <arn>

Azure & GCP: Audit key configurations using Az PowerShell (Get-AzKeyVaultKey) or gcloud (gcloud kms keys list).

Access & Evidence
Access Tier Evidence Captured PQC Relevance
Tier 4 (Cloud IAM Read) Key ARNs, KeySpec (RSA_2048, ECC_NIST_P256), Key usage types, Key origin, Rotation policies, key policies. High. Cloud vendors are rollling out native PQC key types (e.g. ML-KEM).
3.14 Network Devices
Layer 1: Network / Firmware
Why it matters Routers, switches, firewalls, and load balancers run administrative SSH, HTTPS, and tunnel encryption (MACsec). Legacy firmware remains constrained.
Discovery Methods

Probes & Configs: Audit devices using SSH scans, admin TLS scans, SNMP queries (snmpwalk), or query configurations centrally using config archives (Oxidized, Batfish).

Access & Evidence
Access Tier Evidence Captured PQC Relevance
Tier 2 (Network Admin Config Read) Vendor/Model/Firmware details, device SSH ciphers, IPsec parameters, TLS management certs. Medium. Network hardware has long lifecycles. vendor roadmaps dictate path.
3.15 CMDB
Layer 5: Operational
Why it matters A CMDB is the canonical database of Configuration Items (CIs) and relationships. A technical vulnerability finding is unactionable without owning teams, criticality scores, and business service maps.
Discovery Methods

Query schema tables such as cmdb_ci (ServiceNow, BMC Helix, NetBox) for server classes, networks, databases, web servers, CAs, and their associated relationship bindings (cmdb_rel_ci).

POPULATING CONNECTORS LOGIC:
WMI (Windows) + SSH (Linux) + SNMP (Network) = Most Infrastructure Assets
Access & Evidence
Access Tier Evidence Captured PQC Relevance
Tier 1 (CMDB API Read User) CI counts by class, Owner records, business service maps, last discovery timestamps, relationship maps. Critical. Connects scanner-detected OIDs to enterprise owners and risk scoring.

Inventory fields: ci_class, ci_count, owner, business_service, last_discovered, relationships[], crypto_relevance, pqc_owner

Part 4 — Scanner Architecture
Designing a scalable, versioned, and modular enterprise discovery system
4.1 Purpose & 4.2 Architectural Goals

The playbooks in Part 3 produce evidence. The scanner in Part 4 turns that evidence into a continuously updated view of the cryptographic posture of the entire enterprise. It is a system, not a script, built around these core goals:

Coverage
Audit every asset identified in the CMDB target registers, plus actively sweep network blocks to locate shadow deployments.
Repeatability
Ensure playbooks execute deterministically, producing comparable results across different cycles and runs.
Tier-awareness
Enforce strict boundaries so that workers call credentials matching the lowest access tier required.
Idempotency
Re-running playbooks on unchanged configurations produces identical hashes and normalized values.
Auditability
Sign and timestamp all evidence, including execution environment metadata, parameters, and collector signatures.
Composability
Decouple connectors and runners so that adding new playbooks requires zero changes to core pipeline code.
4.3 High-level Architecture

The overall scanner topology flows from ingestion to reporting, decoupling network-heavy workers from the centralized risk analysis engines:

Ingest Layer
CMDB, AWS, Azure, & NetBox Connectors
↓
Asset Graph
Unified relationship model between corporate CIs
↓
Discovery Engine
Playbook selector, credential-vault wrapper & workers
↓
Evidence Store
Signed and versioned append-only observations database
↓
Cryptographic Analysis Engine
Algorithm normalization, PQC mapping & composite detection
↓
Risk Engine
Exposure maps & Composite grading (Mosca's Theorem)
↓
Output Layer
Dashboard UI, Compliance Export, & API
4.4 Component Design

Ingest Layer: Syncs CIs daily from ServiceNow, cloud setups (AWS Config/Azure Resource Graph), and endpoint platforms. Outputs a queryable dependency graph.
Discovery Engine: Selecting playbooks based on target classifications, retrieving scoped credentials from vault (HashiCorp, CyberArk), and running workers.
Cryptographic Analysis: Normalizing raw strings, flagging hybrid composite markers (e.g. rsa-pss-ml-dsa), and querying deprecation databases.
Risk Engine: Joining business tiers with cryptographic features to assign composite priority maps.
Dashboard: Executive and operational widgets showing volume of algorithms, PQC readiness rates, top backlogs, and scan coverage metrics.

EVIDENCE STORE RECORD SCHEMA (JSON):
{ "evidence_id": "uuid-v4-hash-string", "asset_id": "ci-12345", "playbook": "3.2 SSH", "playbook_step": "host_key_algorithm", "tool": "ssh -Q key", "tier": 1, "collected_at": "2026-06-01T11:00:00Z", "collector": "ansible-2026-06-01-12-00-00", "raw": "ssh-ed25519", "normalized": { "algorithm": "Ed25519", "key_size": null, "curve": null, "pqc_relevant": false }, "signature": "<signature-hash>" }
4.5 - 4.8 Operations & Failure Handling

Scheduling & Pacing: Endpoints are scanned on a rolling 7-day schedule, edge portals every 24 hours, and cloud setups dynamically on resource creation signals. Throttlers ensure network limits are respected.
Access-Tier Enforcement: Workers log matching credential aliases used and immediately throw runtime errors if playbook configurations request access tiers higher than the target class policy limits.
Failure & Provenance: Failed commands write a failure record to the Evidence Store rather than crashing, keeping run metrics complete. If credentials fail validation, they are quarantined and trigger alerts.

Part 5 — PQC Migration
Operationalizing scanner outputs into a corporate migration roadmap
5.1 Purpose & 5.2 The 9 Migration Stages

Scanner findings must drive structured remediation. The migration process utilizes a 9-stage workflow to coordinate algorithm uplift across application owners:

1
Find
Filter active findings from the Cryptographic Analysis Engine. Exclude systems slated for decommission or already assigned to active project workstreams.
2
Own
Resolve targets to application or systems owners using CMDB CI records. Escalate to service owners if individual server assignments are missing.
3
Map
Build upstream/downstream maps. If a certificate changes, identify which clients, APIs, or relying party integrations are impacted.
4
Prioritise
Score findings using risk factors: HNDL threat sensitivity, system exposure, business criticality, and regulatory deadlines.
5
Recommend
Draft target PQC algorithms based on current standards (see mapping below) and IETF transition protocols.
6
Pilot
Execute migrations in staging. Verify compatibility of composite/hybrid implementations, network overheads, and library stability.
7
Roll out
Deploy changes via standard change controls, migrating assets in batches grouped by business service to limit blast radius.
8
Track
Log states in the centralized PQC registry (planned, pilot, rolled out, verified). Maintain trace links for compliance metrics.
9
Re-scan
Re-run playbooks against the migrated asset. The new evidence record updates the database and moves the registry state to verified.

5.3 & 5.4 Reporting & Boundaries

The program generates two primary monthly reports: Executive reports tracking total PQC readiness rates, top backlogs, and coverage; and Operational reports listing specific team backlogs.

Scope Limit: The scanner only discovers, recommends, and verifies. Rebuilding applications, configuring appliances, or rekeying databases is owned by systems owners. A migration is only complete when the scanner verifies the updated algorithms in evidence.

Terminology, schemas, and indices
Appendix A — Glossary
Term Definition
Asymmetric Cryptography Public-key cryptography. Relies on mathematical relations between public/private keys. Vulnerable to Shor's algorithm.
CNG Cryptography API: Next Generation. The modern cryptographic service provider interface for Windows OS.
CNSA 2.0 NSA's Commercial National Security Algorithm Suite 2.0. Defines timeline rules for national security assets.
Evidence A signed, normalized observation data record collected during execution of a playbook step.
HNDL Harvest Now, Decrypt Later. Collecting encrypted network traffic today to decrypt once cryptanalytically relevant quantum computers exist.
KEX Key Exchange. Protocol handshake steps that negotiate symmetric keys over open channels.
ML-DSA Module-Lattice-Based Digital Signature Algorithm (FIPS 204). Replaces classical signatures.
ML-KEM Module-Lattice-Based Key-Encapsulation Mechanism (FIPS 203). Replaces classical key exchanges.
PCR Platform Configuration Register. TPM registers recording cryptographic hashes of UEFI boot stages.
PQC Post-Quantum Cryptography. Algorithms resistant to quantum computer attacks.
SLH-DSA Stateless Hash-Based Digital Signature Algorithm (FIPS 205). Standard fallback signature algorithm.
Trust Anchor Root certificates or keys explicitly trusted by the local configuration.
UEFI Unified Extensible Firmware Interface. The standard firmware platform replacing classical BIOS.
