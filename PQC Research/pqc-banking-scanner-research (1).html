How to score each scanned asset · Derived from FS-ISAC Future State and NIST IR 8547

Step 2 — Data Sensitivity Classification (HNDL Risk)
Harvest Now, Decrypt Later (HNDL) Even though quantum computers do not exist today, adversaries are recording encrypted traffic now and storing it to decrypt later. This means data being transmitted today under RSA/ECDH is already at risk. Assets protecting long-lived sensitive data must be treated as Tier 1.

NIST Asset Reporting Format — CBOM
Cryptography Bill of Materials · CycloneDX v1.7 · ECMA-424 · Endorsed by NIST SP 1800-38B
Standard NIST SP 1800-38B endorses the CycloneDX CBOM (Cryptography Bill of Materials) as the standard output format for cryptographic asset discovery. CBOM is now ratified as ECMA-424. Every finding from the scanner should be output as a CBOM record.
CBOM Document Envelope

Every CBOM report must include this top-level structure:

Minimum Required Fields (per NIST SP 1800-38B)
Every reported asset must include at minimum: (1) Algorithm name and specific variant — not just "RSA" but "RSA-2048"  ·  (2) Location where it was found (host, service, file path)  ·  (3) Usage context (TLS, signing, encryption at rest, key exchange)  ·  (4) Owning system or application  ·  (5) PQC vulnerability flag — is this algorithm quantum-vulnerable per NIST IR 8547?  ·  (6) Deprecation deadline — when it must be replaced (2030 or 2035).

Scanner Connection Architecture
How your application connects to each component · 5 connection mechanisms · Derived from NIST SP 1800-38B
NIST SP 1800-38B defines two top-level strategies Active (your scanner initiates contact) and Passive (your scanner observes traffic). These break down into five specific connection mechanisms below. Most banking environments will use all five simultaneously.
Mechanism 1 — Active Network Handshake
No credentials required · Zero production impact · Agentless

Your scanner initiates a real protocol handshake — TLS ClientHello, SSH KEX Init, IKEv2 SA Init — and reads back what the server advertises. The connection is never completed (aborted after ServerHello). Requires only network reachability to the target IP and port.

Libraries: SSLyze (Python), testssl.sh, ssh-audit, pqcscan (Rust), Python ssl module, Java SSLEngine.

Mechanism 2 — Passive Network Tap / SPAN
No scanner-server contact · Zero production impact · Sees actual negotiated ciphers

Your scanner sits on a network tap (physical) or SPAN/mirror port and captures all traffic. TLS ClientHello and ServerHello are not encrypted — cipher suites and certificates are fully visible without decrypting payload. Catches actual negotiated ciphers in production (what's really used, not just what's configured).

Deployment: Cisco SPAN, Gigamon taps, network broker appliances. Libraries: Wireshark/tshark, Zeek (Bro), SandboxAQ AQtive Guard (commercial), Python Scapy/pyshark.

What You SeeAdvantage Over Active Scan

Mechanism 3 — Authenticated Agent / SSH + WinRM Remote Execution
Requires privileged service account · Low production impact · Deepest visibility

Your scanner authenticates to each server via SSH (Linux) or WinRM (Windows) and runs commands remotely to enumerate crypto assets on the filesystem and in OS configuration. Use a read-only service account provisioned through your PAM system.

Libraries: Python paramiko (SSH), pywinrm (WinRM), Ansible (orchestrated multi-host). Can be replaced by a lightweight deployed agent on highest-priority systems.

Linux — Remote Commands via SSH
# Find all crypto key and certificate files find / -name "*.pem" -o -name "*.key" -o -name "*.crt" \ -o -name "*.p12" -o -name "*.jks" -o -name "*.pfx" 2>/dev/null # Check OpenSSL version and supported algorithms openssl version openssl ciphers -v 'ALL' # Check SSH server configuration grep -E 'HostKey|KexAlgorithms|Ciphers|MACs' /etc/ssh/sshd_config # List installed crypto libraries dpkg -l | grep -E 'openssl|libssl|libgcrypt|bouncycastle' rpm -qa | grep -E 'openssl|nss|libgcrypt' # Check Java keystores find / -name "cacerts" -o -name "*.jks" 2>/dev/null keytool -list -keystore /path/to/keystore.jks
Windows — Remote Commands via WinRM (PowerShell)
# Enumerate Windows certificate store Get-ChildItem -Path Cert:\LocalMachine\ -Recurse | Select-Object Subject, Issuer, NotAfter, SignatureAlgorithm, Thumbprint # Check Schannel (TLS) registry settings Get-Item "HKLM:\SYSTEM\CurrentControlSet\Control\SecurityProviders\SCHANNEL" # Find certificate files on filesystem Get-ChildItem -Path C:\ -Recurse ` -Include *.pfx,*.p12,*.pem,*.cer,*.crt -ErrorAction SilentlyContinue # Check Kerberos encryption types Get-ItemProperty "HKLM:\SYSTEM\CurrentControlSet\Control\Lsa\Kerberos\Parameters"
Mechanism 4 — API / Management Console (Vendor-Specific)
Authenticated API calls · Read-only permissions · Zero production impact

Different protocol for each component type. All use read-only authenticated access to vendor management interfaces.

4A — HSM (PKCS#11 Interface)
Protocol: PKCS#11 C API via vendor-provided shared library (.so/.dll). Thales Luna → CKS library · Entrust nShield → nfast library. Enumerates key metadata only — no private key export by design.
import PyKCS11 lib = PyKCS11.PyKCS11Lib() lib.load('/usr/lib/libCryptoki2_64.so') # Thales Luna session = lib.openSession(slot) session.login(PyKCS11.CKU_USER, hsm_pin) # Enumerate all objects in HSM partition objects = session.findObjects() for obj in objects: attrs = session.getAttributeValue(obj, [ PyKCS11.CKA_CLASS, # CKO_PRIVATE_KEY, CKO_CERTIFICATE... PyKCS11.CKA_KEY_TYPE, # CKK_RSA, CKK_ECDSA, CKK_AES PyKCS11.CKA_MODULUS_BITS, # RSA key size in bits PyKCS11.CKA_LABEL, # Key label/name PyKCS11.CKA_ID # Key identifier ])
4B — Key Management Server (KMIP Protocol)
Protocol: KMIP 1.x/2.x over mutual TLS (port 5696). Supported by: Thales CipherTrust, IBM SKLM, HashiCorp Vault, Entrust KeyControl.
# KMIP Locate + Get Attributes — enumerate all managed keys request = { "operation": "Locate", "attributes": [{"name": "Object Type", "value": "Symmetric Key"}] } # Returns key IDs — then Get Attributes for each: # Algorithm, Key Length, Cryptographic Usage Mask, State, Creation Date
4C — Cloud KMS (REST API)
# AWS KMS — list keys and describe each GET https://kms.{region}.amazonaws.com → ListKeys, DescribeKey # Returns: KeyAlgorithm (RSA_2048, EC_SIGN_P256 — vulnerable; AES_256 — safe) # Azure Key Vault GET https://{vault}.vault.azure.net/keys → list, getKey # GCP Cloud KMS GET https://cloudkms.googleapis.com/v1/projects/{p}/keyRings/{kr}/cryptoKeys # Auth: IAM read-only role / service principal — no write permissions needed
4D — Microsoft ADCS / Active Directory (LDAP + certutil)
# Query all issued certificates from Microsoft ADCS certutil -view -restrict "Disposition=20" \ -out "CommonName,NotAfter,CertificateTemplate,PublicKeyAlgorithm" # LDAP query — enumerate CA objects ldapsearch -H ldaps://dc.bank.local:636 \ -b "CN=Public Key Services,CN=Services,CN=Configuration,DC=bank,DC=local" \ "(objectClass=pKIEnrollmentService)"
4E — Database TDE (JDBC SQL Queries)
-- Oracle TDE status — requires read-only DBA account SELECT * FROM V$ENCRYPTION_WALLET; SELECT OWNER, TABLE_NAME, COLUMN_NAME, ENCRYPTION_ALG FROM DBA_ENCRYPTED_COLUMNS; -- SQL Server TDE SELECT db.name, dek.encryption_state, dek.encryptor_type, dek.key_algorithm, dek.key_length FROM sys.dm_database_encryption_keys dek JOIN sys.databases db ON dek.database_id = db.database_id;

# Get all TLS secrets across namespaces — read-only RBAC token kubectl get secrets --all-namespaces -o json | \ jq '.items[] | select(.type=="kubernetes.io/tls") | .metadata.name' # Get etcd encryption config kubectl get secret -n kube-system -o yaml | grep -A5 'encryption' # Get API server TLS certificate kubectl get secret -n kube-system kube-apiserver-certs -o yaml
Mechanism 5 — Static Analysis / Source Code Scan (SAST)
Read access to repository or filesystem · No live system connection · Finds hardcoded algorithms

Directly reads source code, compiled binaries, and dependency manifests without connecting to any live system. Uses pattern matching and AST analysis to find hardcoded algorithm names, crypto API calls, and vulnerable library versions.

Tools: Semgrep (with crypto rulesets), Veracode, Checkmarx, CycloneDX SBOM generators (Syft, cdxgen), custom tree-sitter AST parsers.

Flagged patterns — Python
from Crypto.PublicKey import RSA # pycryptodome RSA — VULNERABLE import ssl; ssl.PROTOCOL_TLSv1 # deprecated TLS version — VULNERABLE hashlib.md5(data) # MD5 usage — VULNERABLE cipher = DES.new(key, DES.MODE_ECB) # 3DES usage — DISALLOWED NOW
Flagged patterns — Java
KeyPairGenerator.getInstance("RSA") // RSA key generation — VULNERABLE Cipher.getInstance("DES/ECB/PKCS5Padding") // 3DES — DISALLOWED NOW MessageDigest.getInstance("SHA-1") // SHA-1 — DISALLOWED NOW

Agent-Based vs Agentless — Key Design Decision
ApproachMechanismsProsConsNIST Recommendation
Agentless 1 (Active), 2 (Passive), 4 (API) No software deployment on target hosts; easier change management approval; lower operational overhead Misses memory-resident keys, in-process crypto calls, some embedded crypto usage Start here — provides coverage and scope without deployment friction
Agent-based 3 (SSH/WinRM), 5 (SAST) Deeper access — finds process-level crypto, memory-resident keys, hardcoded algorithm strings Requires deployment approval for every host; higher operational overhead Deploy on highest-risk systems (CBS, HSM hosts, payment switches) after agentless pass
