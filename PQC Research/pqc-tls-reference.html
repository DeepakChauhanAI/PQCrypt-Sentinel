The Common Misconception
Why "upgrade to TLS 1.3 and you're safe" is wrong — and what the scanner actually needs to check
The wrong mental model "Older TLS versions used cryptographically vulnerable encryption, and newer versions don't." This is a common but slightly wrong way to think about it. Old TLS versions were deprecated for two entirely different reasons — and only one of them is fixed by upgrading the TLS version.
Two Separate Vulnerability Stories

The deprecation of TLS 1.0 and 1.1 happened for one reason. The PQC migration of TLS 1.3 is happening for a completely different reason. They are separate problems, requiring separate fixes. A PQC scanner needs to track both independently.

Reason for deprecation / concernFixed by upgrading TLS version?What fixes it?
Protocol design weaknesses — how the protocol handled padding, compression, renegotiation (BEAST, POODLE, CRIME, ROBOT attacks) Yes TLS 1.2 and TLS 1.3 redesigned the protocol. These attacks are not possible against a correctly implemented TLS 1.3 server.
Weak or quantum-vulnerable public-key algorithms — RSA, ECDSA, DH, ECDH used inside the handshake No Replacing the algorithms inside the handshake with PQC equivalents (ML-KEM for key exchange, ML-DSA for signatures). TLS version is irrelevant to this problem.
The critical implication for the scanner A TLS 1.3 endpoint using X25519 for key exchange will pass every classic TLS security check — correct version, strong cipher, perfect forward secrecy. It will still be flagged as quantum-vulnerable by the PQC scanner, because X25519 (ECDH) is broken by Shor's algorithm. TLS version and PQC readiness are orthogonal measurements. The scanner must check both.
The Protocol Attack Family (Story A)

These are the named attacks that killed TLS 1.0 and 1.1. They are all protocol-level weaknesses — problems with how the handshake was designed, not with the underlying math of RSA or ECDH. Moving to TLS 1.2 or 1.3 closes all of these.

AttackYearWhat it exploitedTLS versions affected
BEAST2011CBC-mode encryption with predictable IVs in TLS 1.0 — allowed plaintext recoveryTLS 1.0 only
POODLE2014SSL 3.0's CBC padding handling — forced downgrade to SSL 3.0, then exploitedSSL 3.0 (POODLE v2 hit TLS 1.x)
CRIME2012TLS compression — compressed data leaked info about the plaintext via size differencesTLS 1.0, 1.1, 1.2 with compression
ROBOT2017RSA PKCS#1 v1.5 padding oracle in TLS 1.2 implementations — allowed RSA decryptionTLS 1.2 implementations (not the protocol itself)
DROWN2016SSLv2 export cipher suites — servers supporting both SSLv2 and TLS 1.x were vulnerableServers with SSLv2 enabled
SWEET32201664-bit block ciphers (3DES) — birthday attacks on long sessionsAny TLS version using 3DES
The Public-Key Algorithm Problem (Story B — PQC)

Inside every TLS handshake, regardless of version, there are three categories of public-key operations. Every standard algorithm currently used in all three categories is broken by Shor's algorithm on a sufficiently powerful quantum computer.

OperationPurposeClassical algorithms in usePQC replacement
Key Exchange How both sides agree on a shared session key without ever sending it over the network RSA key transport, (EC)DHE, ECDH, X25519 — all quantum-vulnerable ML-KEM (FIPS 203) — as standalone or hybrid with X25519
Server Authentication Proof that the server really is who the certificate says it is — prevents impersonation RSA-PSS, ECDSA, Ed25519 — all quantum-vulnerable ML-DSA (FIPS 204), SLH-DSA (FIPS 205), Falcon
Client Authentication (optional mTLS) Proof that the client holds a specific private key — used in mTLS, device auth, smartcards Same as server auth — RSA-PSS, ECDSA — quantum-vulnerable ML-DSA, SLH-DSA
What is NOT quantum-vulnerable in TLS AES-128-GCM, AES-256-GCM, ChaCha20-Poly1305 (bulk data encryption) — all safe. SHA-256, SHA-384 (hash / HMAC) — safe at current sizes. HKDF key derivation — safe. The quantum vulnerability is exclusively in the public-key operations in the handshake phase. Once the session key is established via a safe key exchange, the data transmission is quantum-safe.

What RFC 8996 (2021) Actually Did

RFC 8996 — "Deprecating TLS 1.0 and TLS 1.1" — made these versions formally prohibited in the IETF standards track. Before 2021, they were simply "not recommended." After 2021, offering TLS 1.0 or 1.1 is a standards violation. Every major browser removed support in 2020–2021. The scanner should treat any endpoint advertising TLS 1.0 or 1.1 as a distinct finding category: deprecated protocol version, separate from the PQC algorithm finding.

The scanner reports TLS version as a separate finding field, not just as part of the cipher suite. Every finding record carries both the version and the key exchange algorithm. This matters because the remediation path differs:

The TLS 1.2 Handshake Difference

TLS 1.2 has a 2-RTT handshake and slightly different structure. The key difference relevant to the scanner: in TLS 1.2, RSA key transport is possible — the client generates the session key and encrypts it with the server's RSA public key. There is no forward secrecy. This is the worst-case scenario for HNDL: an attacker who recorded the TLS 1.2 session only needs to break the server's RSA public key (not an ephemeral key) to decrypt everything. The scanner flags TLS_RSA_* cipher suites distinctly from TLS_ECDHE_RSA_* because the HNDL exposure is fundamentally different.

PQC in TLS — The Full Picture
HNDL threat, hybrid schemes, pure PQC, signature algorithms, and real-world deployment status
Why TLS 1.3 Is Still Quantum-Vulnerable

TLS 1.3 is the most secure protocol version ever standardised. The public-key algorithms used inside it are quantum-vulnerable. These are two separate facts that coexist. A TLS 1.3 server using X25519 for key exchange will pass every classic security audit — correct version, strong AEAD cipher, perfect forward secrecy, no deprecated features. It will still be broken by a sufficiently powerful quantum computer.

The HNDL Threat — Why This Matters Now

The quantum computer that breaks TLS doesn't exist yet. The threat is real now because of the Harvest Now, Decrypt Later (HNDL) attack pattern:

1
Attacker records today's TLS traffic
Nation-state adversaries passively capture TLS sessions at internet exchange points, ISP taps, or cable intercepts. The captured traffic is useless today — they cannot decrypt it.
TLS 1.3 session: ClientHello (X25519 pubkey), ServerHello (X25519 pubkey + ciphertext), encrypted records
2
Attacker stores the ciphertext
The captured handshake + encrypted session data is archived. Storage is cheap. A terabyte of captured TLS sessions costs dollars to store indefinitely. This is already happening.
3
Quantum computer arrives (est. 5–15 years)
A cryptographically relevant quantum computer (CRQC) runs Shor's algorithm against the stored X25519 public keys from the recorded handshakes.
Shor's algorithm breaks X25519 discrete log → recovers the shared secret → derives the session key
4
Attacker decrypts all stored sessions
With the session key recovered, every encrypted record from that TLS session is decrypted. Any data transmitted under that session — API calls, authentication tokens, sensitive documents, private communications — is now readable.
The central business argument for PQC migration of TLS If the data protected by a TLS session has a confidentiality requirement longer than the time until a quantum computer exists, that data is already compromised. For any organisation transmitting data that must remain confidential for 5+ years — trade secrets, long-term contracts, health records, identity data — the HNDL window is already open. Migration cannot wait.
Hybrid PQC Key Exchange — The Current Standard

The industry solution for HNDL mitigation without breaking compatibility is hybrid key exchange: run a classical algorithm (X25519) and a PQC algorithm (ML-KEM-768) in parallel in the same TLS connection. The session key is derived from both. The connection is secure if either algorithm holds.

Hybrid named groupComponentsIANA codeStatus
X25519MLKEM768 X25519 (classical ECDH) + ML-KEM-768 (FIPS 203 PQC KEM) 0x639A Current default — Chrome, Firefox, Cloudflare, AWS. Over 52% of TLS 1.3 traffic as of 2025.
X25519Kyber768 X25519 + Kyber-768 (pre-standard name — Kyber was renamed to ML-KEM) 0x4588 Legacy draft name — still seen in the wild. Scanner should recognise and map to ML-KEM.
P256MLKEM768 NIST P-256 (ECDH) + ML-KEM-768 0x639C Less common — alternative for environments that require P-256.

PQC Signature Algorithms in TLS

Key exchange is only half the picture. Server authentication — proving the server's identity via its certificate — also uses public-key cryptography. The server's certificate contains an RSA or ECDSA public key, and the CA that signed it used RSA or ECDSA. Both are quantum-vulnerable. The PQC replacements for TLS authentication are:

AlgorithmStandardFormer nameSecurity basisNote
ML-DSA-44, ML-DSA-65, ML-DSA-87FIPS 204CRYSTALS-DilithiumModule lattice (MLWE + SelfTargetMSIS)Primary PQC signature standard. Larger signatures than RSA/ECDSA — check protocol size limits.
SLH-DSA (multiple parameter sets)FIPS 205SPHINCS+Hash functions onlyConservative backup. Security relies only on hash functions — simpler assumptions. Slower, larger. Use for root CAs and long-lived signing keys.
Falcon-512, Falcon-1024NIST Round 4FalconNTRU latticeSmaller signatures than ML-DSA. Complex implementation. Not yet widely deployed.
Real-world PQC deployment status (2026) Key exchange (ML-KEM hybrid) is deployed at scale — Chrome, Firefox, Cloudflare, AWS, Go 1.24+, OpenSSL 3.5+ all support X25519MLKEM768. Certificate/signature migration (ML-DSA) is not yet deployed at scale — most TLS certificates in production still use RSA or ECDSA. This means the scanner will find: key exchange = hybrid safe, certificate = still RSA/ECDSA vulnerable. Both findings are correct and both need remediation on different timelines.
Per-Component TLS — Why One Scan Per Host Is Not Enough
TLS is not an OS feature — every application bundles its own library, and they don't share state
The key insight TLS is not implemented by the operating system. It is implemented by a library that each application bundles, links against, or ships with. Different applications on the same host use completely separate TLS stacks with different versions, different cipher support, and different PQC readiness. A single "host is PQC-ready" finding is meaningless — it has to be per-component.

Three Concrete Implications
Same host, different PQC readiness

A host running Nginx (OpenSSL 3.5) and a Java app (JDK 21): Port 443 (Nginx) supports X25519MLKEM768 — PQC-ready. Port 8443 (Java) supports only x25519 and secp256r1 — quantum-vulnerable. Both are on the same machine. Both are part of the same business service. They have completely different PQC exposure. The scanner must report per-port, not per-host.

Libraries are pinned at build time, not runtime

Most production apps do not dynamically reload their TLS library. The library version is whatever was linked when the binary was built. So even if you upgrade OpenSSL on the host, a Python app that statically links its own OpenSSL (via a wheel) or a Java app running on JDK 21 keeps the old one. Host-level library upgrades are necessary but not sufficient — each application must be rebuilt or restarted against the new library.

The negotiated algorithm is the intersection of both sides

PQC is only negotiated when both client and server support it. A Java microservice (JDK 21, no PQC) calling another service (Go 1.24, PQC-capable) will negotiate X25519 — because the Java client dictates the maximum the connection can use. One slow component holds back the whole call chain.

Blast Radius — Migration Order on a Single Host

You cannot PQC-upgrade a host. You have to PQC-upgrade each component, in dependency order, against the dependencies of the things it talks to.

1
Upgrade host-level OpenSSL to 3.5+
This unlocks PQC for all components that dynamically link the system OpenSSL: nginx, curl, wget, Python (non-vendored), PostgreSQL's libpq.
2
Rebuild and redeploy every Python app that statically links OpenSSL
Apps using vendored OpenSSL (via pip wheels) must be rebuilt with updated dependencies. pip install --upgrade cryptography pyOpenSSL then redeploy.
3
Upgrade JDK from 21 to 24 for every Java app
JDK upgrade is a significant change — test thoroughly. Every Java app on the host must be tested against JDK 24.
4
Upgrade Node.js from 18 to 24 for every Node app
Node 18 is the current LTS but does not support X25519MLKEM768. Node 24 is required.
5
Rebuild Go binaries with Go 1.24+
Go compiles the TLS library into the binary. Each Go service must be recompiled with Go 1.24+. The Go runtime cannot be updated without recompiling.
6
Restart services in dependency order
Databases first (they are depended upon), then internal services, then edge proxies last. If any one step is missed, the negotiated cipher falls back to classical X25519.

A scanner that understands the per-component reality produces output like this — one line per port per host, not one line per host:

Why TLS Is Significant for the Scanner Project
Five specific reasons TLS is the foundation of a PQC scanner — not just a feature
The bottom line up front TLS is not just a feature in the PQC scanner. It is the foundation, the proof of value, the largest data source, the easiest to scan, and the only place where PQC is observably deployed today. If the scanner had to be cut down to its absolute essentials, TLS scanning, CBOM generation from the results, and a hybrid-PQC-aware handshake classifier are the three capabilities that would still make it useful.
Reason 1 — TLS Is the Largest Public-Key Crypto Surface in Any Organisation

Every web service, API, microservice call, database connection (when encrypted), load balancer, API gateway, CDN, mail server, VPN, and most internal service-to-service traffic uses TLS. A typical mid-size enterprise has tens of thousands of TLS endpoints. There is no other crypto surface that comes close in volume.

Code signing certificates number in the dozens. Root CAs number in single digits. SSH keys number in thousands. TLS endpoints number in tens of thousands for even a moderate-sized organisation. The scanner that does TLS well immediately covers the overwhelming majority of the cryptographic attack surface.

Reason 2 — TLS Is the Only Place Where PQC Is Deployed at Scale Today

PQC for code signing, document signing, S/MIME email, firmware updates, and PKI certificates is still research and pilot. But PQC for TLS already works, ships in browsers, ships in CDNs, and is deployed at massive scale:

TLS is the only place where a PQC scanner can produce real, measurable, observable findings today — not forecasts, not readiness assessments, but actual "this endpoint supports X25519MLKEM768 and this one does not." Everything else is prediction. TLS is the present.

Reason 3 — TLS Is the Lowest-Friction Discovery Source

For every other layer (SBOM, CA, cloud KMS, HSM), the scanner needs credentials, integration, and explicit access grants. For TLS, it only needs a network path to port 443. An active handshake is the simplest possible probe — it is what curl -v does.

This has three practical consequences: (1) the TLS scanner is the easiest module to deploy — it requires zero customer-side changes; (2) it produces findings within minutes of pointing it at an IP range; (3) it is the first signal that proves the tool works, which is critical for customer onboarding and internal demos. The scanner's most reliable, highest-coverage, lowest-permission data source is "send a ClientHello, read the ServerHello."

Reason 4 — TLS Is What an Attacker Would Break First

In the HNDL threat model, the most valuable traffic to record and break later is TLS — because that is where long-lived sensitive data flows in motion. An attacker is not recording raw database TDE ciphertext or stored PGP-encrypted files; they are recording TLS sessions that carry the same data as it moves between services, from user to application, from application to database.

This means a customer's CFO will fund PQC remediation for TLS long before they fund it for legacy code signing or HSM firmware upgrades. The scanner that surfaces "you have 4,217 TLS endpoints, 3,891 of which are quantum-vulnerable, of which 412 are HNDL-exposed based on data classification" is the one that gets the budget conversation started.

Reason 5 — TLS Scanning Is the On-Ramp to the Rest of the Inventory

Once the scanner does TLS well, every other layer extends the same inventory model. The CBOM built from TLS data — certificates, algorithms, protocols, dependencies — is the same CBOM enriched with CA data, SBOM data, and KMS data. The relationship graph built from TLS endpoints and their certificate chains is the same graph extended with software and business service relationships.

A scanner that nails TLS first has a working, demonstrable product by month 3. A scanner that tries to do all 7 layers simultaneously has nothing demonstrable by month 3. TLS-first is the only realistic MVP path, and TLS is the substrate that the rest of the product grows on.

RSA (first position)
RSA key transport — the client generates the pre-master secret and encrypts it directly with the server's RSA public key. There is NO forward secrecy. If the server's private RSA key is ever broken — by a quantum computer, by theft, by a future vulnerability — every session ever recorded with this key can be decrypted retroactively. This is the worst-case HNDL scenario.
Worst case — no PFS
RSA key transport vs ECDHE — different HNDL exposure The scanner must distinguish these. TLS_RSA_* (no PFS) means an attacker only needs to break the long-term RSA server key — once — to decrypt all past sessions that used that key. TLS_ECDHE_RSA_* (PFS via ephemeral ECDH) means the attacker must break a unique ephemeral key per session. Both are quantum-vulnerable, but TLS_RSA_* is categorically worse and should be flagged as a separate, more urgent finding.

Glossary — Key Technical Terms
Every term used in TLS and PQC scanning — defined precisely
Protocol & Standard Terms
TLS
Transport Layer Security. The current standard protocol for encrypted network communication, successor to SSL. Versions: 1.0 (deprecated), 1.1 (deprecated), 1.2 (still in use), 1.3 (current standard, RFC 8446).
SSL
Secure Sockets Layer. The predecessor to TLS. SSL 2.0 and 3.0 are completely broken and formally prohibited. In everyday speech, "SSL" is often used as a synonym for TLS — the HTTPS padlock in a browser uses TLS, not SSL.
RFC 8446
The IETF standard that defines TLS 1.3 (2018). Specifies the protocol redesign: mandatory AEAD ciphers, removal of RSA key transport, 1-RTT handshake, and the named groups extension that enables PQC key exchange.
RFC 8996
The IETF standard that formally deprecated TLS 1.0 and TLS 1.1 (2021). Before RFC 8996 they were "not recommended." After RFC 8996 they are prohibited. The scanner flags any endpoint advertising these versions.
Cipher suite
A named combination of cryptographic algorithms negotiated during the TLS handshake. In TLS 1.2: encodes key exchange + auth + bulk cipher + MAC in a single string. In TLS 1.3: encodes only bulk cipher + hash — key exchange is in the named group extension separately.
Named group
In TLS 1.3, the discrete set of key exchange algorithms offered (ClientHello) or chosen (ServerHello) via the supported_groups and key_share extensions. Examples: x25519, secp256r1, X25519MLKEM768. This is where PQC lives in TLS 1.3.
Forward secrecy (PFS)
Perfect Forward Secrecy — a property where each session uses a unique ephemeral key that is discarded after use. If the server's long-term private key is later compromised, previously recorded sessions cannot be decrypted. Required in TLS 1.3; optional in TLS 1.2.
AEAD
Authenticated Encryption with Associated Data. A cipher mode that simultaneously encrypts and provides integrity verification. AES-GCM and ChaCha20-Poly1305 are AEAD ciphers. TLS 1.3 mandates AEAD-only cipher suites. Older CBC-mode ciphers (which enabled BEAST/POODLE) are removed.
SNI
Server Name Indication. A TLS extension in ClientHello that tells the server which hostname the client wants to connect to. Visible in plaintext even in TLS 1.3 (before the encrypted session is established). The scanner reads SNI to confirm which hostname is being scanned.
Handshake & Key Exchange Terms
ClientHello
The first message in a TLS handshake, sent by the client. Lists all cipher suites and named groups the client supports. The scanner sends a custom ClientHello advertising every PQC named group to maximise the server's response information.
ServerHello
The server's response to ClientHello. Contains the single cipher suite and named group the server chose. The scanner's primary data source — the chosen named group reveals the key exchange algorithm actually used.
Key exchange
The step where both sides agree on a shared session key without transmitting it. Classical: RSA (no PFS), ECDH, DHE. PQC: ML-KEM (KEM = Key Encapsulation Mechanism). Hybrid: X25519MLKEM768 = classical + PQC in parallel.
KEM
Key Encapsulation Mechanism. The type of key exchange used by ML-KEM (PQC). Unlike DH (both sides contribute), in a KEM: the client generates a random key, encapsulates it under the server's public key, and the server decapsulates it. The result is a shared secret neither side transmitted.
ECDHE
Elliptic Curve Diffie-Hellman Ephemeral. Classical key exchange — both sides generate an ephemeral (one-time) EC key pair, exchange public keys, compute the same shared secret via the DH protocol. Provides forward secrecy. Quantum-vulnerable via Shor's algorithm.
RSA key transport
TLS 1.2-only mechanism: the client generates the pre-master secret and encrypts it directly with the server's RSA public key. No forward secrecy — if the server's private key is ever broken, all past sessions decrypt. Removed in TLS 1.3. Flagged as the most urgent finding by the scanner.
HKDF
HMAC-based Key Derivation Function. Used in TLS 1.3 to derive all session keys from the shared secret established during key exchange. Quantum-safe at SHA-256/SHA-384. Not a vulnerability — the scanner notes but does not flag it.
PQC Algorithm Terms
ML-KEM (FIPS 203)
Module-Lattice-based Key Encapsulation Mechanism. The NIST PQC standard for key exchange. Formerly CRYSTALS-Kyber. Three parameter sets: ML-KEM-512 (Level 1), ML-KEM-768 (Level 3, recommended), ML-KEM-1024 (Level 5). Used in TLS as X25519MLKEM768 (hybrid) or MLKEM768 (pure).
ML-DSA (FIPS 204)
Module-Lattice-based Digital Signature Algorithm. The NIST PQC standard for signatures. Formerly CRYSTALS-Dilithium. Three parameter sets: ML-DSA-44 (Level 2), ML-DSA-65 (Level 3, recommended), ML-DSA-87 (Level 5). Used in TLS for server/client authentication certificates.
SLH-DSA (FIPS 205)
Stateless Hash-based Digital Signature Algorithm. The NIST PQC conservative backup signature standard. Formerly SPHINCS+. Security relies only on hash functions — not lattices. Slower and larger signatures than ML-DSA. Used for long-lived signing keys (root CAs, firmware signing).
Hybrid PQC
A scheme that combines a classical algorithm and a PQC algorithm in parallel — the connection is secure if either holds. In TLS: X25519MLKEM768 = X25519 (classical ECDH) + ML-KEM-768 (PQC KEM). The session key is derived from both. The recommended transitional state for TLS key exchange.
IANA named group code
The numeric identifier for a TLS key exchange algorithm registered with IANA and used in the ClientHello/ServerHello key_share extension. X25519MLKEM768 = 0x639A. X25519 = 0x001D. P-256 = 0x0017. The scanner reads these codes directly from the TLS handshake bytes.
OID (Object Identifier)
A globally unique dotted-number identifier for a cryptographic algorithm, embedded in X.509 certificates. sha256WithRSAEncryption = 1.2.840.113549.1.1.11. ML-DSA-65 = 2.16.840.1.101.3.4.3.18. The scanner reads OIDs from certificates and looks them up in its algorithm registry to classify PQC status.
Attack & Threat Model Terms
HNDL
Harvest Now, Decrypt Later. The threat model where adversaries record today's encrypted TLS traffic and store it to decrypt later when a quantum computer is available. The reason PQC migration cannot wait — data being transmitted today under RSA/ECDH key exchange is potentially already compromised.
CRQC
Cryptographically Relevant Quantum Computer. The type of quantum computer powerful enough to run Shor's algorithm at scale and break RSA/ECDH. Does not exist as of 2026. Expert estimates for when it might exist range from 5 to 15 years. The HNDL attack works against future CRQCs using data collected today.
BEAST
Browser Exploit Against SSL/TLS (2011). Exploited predictable IVs in TLS 1.0's CBC-mode encryption to decrypt session data. Fixed by TLS 1.2's changes to CBC record processing and by the move to AEAD ciphers in TLS 1.3.
POODLE
Padding Oracle On Downgraded Legacy Encryption (2014). Forced downgrade to SSL 3.0 and exploited its CBC padding handling. Killed SSL 3.0. POODLE v2 found similar issues in some TLS 1.x implementations.
CRIME
Compression Ratio Info-leak Made Easy (2012). Exploited TLS compression — by observing how ciphertext size changed as an attacker injected data, they could recover plaintext. Fixed by disabling TLS compression entirely, which TLS 1.3 mandates.
ROBOT
Return Of Bleichenbacher's Oracle Threat (2017). Exploited RSA PKCS#1 v1.5 padding in TLS 1.2 implementations — a category of timing-based oracle attack that allows RSA decryption. Affects TLS 1.2 implementations using RSA key transport. Another reason to deprecate RSA key transport.
Scanner & Inventory Terms
CBOM
Cryptographic Bill of Materials. A machine-readable inventory of all cryptographic assets found during scanning — algorithms, keys, certificates, protocols. Standardised in CycloneDX format (ECMA-424). The primary output format of the PQC scanner. NIST SP 1800-38B endorses it.
SBOM
Software Bill of Materials. A machine-readable list of all software components and dependencies in an application. CycloneDX or SPDX format. Used by the scanner to identify which TLS library version an application uses — without active network scanning.
SARIF
Static Analysis Results Interchange Format. The output format for SAST (code scanning) findings. Integrates natively with GitHub Advanced Security, GitLab SAST, and Azure DevOps as pull request annotations.
CycloneDX
An OWASP standard for BOM (Bill of Materials) documents — both SBOM and CBOM use this format. Ratified as ECMA-424. The scanner outputs CycloneDX CBOM JSON as its primary machine-readable format.
Active scanning
The scanner initiates connections to targets — sends TLS ClientHello, reads ServerHello, closes connection. No credentials needed for TLS/SSH. Zero production impact. Tells you what the server is configured to support.
Passive scanning
The scanner observes traffic on a SPAN/mirror port without sending any packets. Tells you what is actually being negotiated in production. Catches legacy clients that downgrade to weaker ciphers — something active scanning may miss.
