Section 1

How to use this guide

If you are new to networking, the fastest way to understand this topic is to read it in three passes:

  1. First, learn the basic vocabulary: host, packet, protocol, certificate, and key.
  2. Second, learn the OSI model, which is a way of grouping network behavior into layers.
  3. Third, connect each enterprise component to the layer where its cryptography appears.

What this scanner is really doing

A post-quantum cryptography scanner is not just looking for "encryption." It is trying to find every place where an organization depends on cryptographic trust: certificates, VPN tunnels, secure web traffic, identity systems, hardware security modules, Wi-Fi authentication, database links, and signing keys.

Section 2

Networking basics, in plain English

What is a network?

A network is a group of devices that can communicate with each other. The devices could be laptops, servers, cloud services, phones, printers, Wi-Fi access points, routers, or security appliances.

What is a packet?

A packet is a small bundle of data sent across the network. Big messages are broken into packets so they can travel efficiently and be reassembled at the destination.

What is a protocol?

A protocol is a set of rules for communication. For example, HTTP tells a browser how to talk to a web server. TLS tells two systems how to create a secure encrypted channel.

What is an endpoint?

An endpoint is any device or service that participates in communication. In security work, the word usually means a user device, server, VM, or container host.

The important idea is that "networking" is not one thing. It is a stack of layers: some layers move bits, some layers move packets, and some layers create trust, identity, and encryption.

Section 3

The OSI model, explained simply

The OSI model is a teaching framework that divides networking into seven layers. You do not need to memorize it as a law of nature. You need it because it helps you answer one question: where does a given technology operate, and where does its crypto live?

Layer Plain meaning Typical examples Why the scanner cares
7. Application The software people directly use. Web apps, APIs, SaaS portals, email, chat, admin consoles. This is where business workflows and user-facing crypto often appear.
6. Presentation How data is encoded, serialized, compressed, or encrypted. JSON, Protobuf, PDF encryption, JWE, codecs, document parsers. Some crypto is hidden inside libraries or data formats instead of visible UI code.
5. Session How two systems keep a conversation going over time. SSO, JWT sessions, gRPC sessions, WebSockets, VPN tunnels. Session identity and token handling are often crypto-dependent.
4. Transport How data moves between two hosts reliably or quickly. TCP, UDP, TLS termination points, load balancers, API gateways. Most secure connections are protected here or just above it.
3. Network How packets find their way across networks. IP, routers, IPsec, VPN gateways, DNSSEC validators. This layer matters when cryptography protects routing, tunneling, or DNS trust.
2. Data Link How devices communicate on the same local network. Ethernet, Wi-Fi, switches, 802.1X, MACsec, ARP. Network access security and Wi-Fi authentication often use certificates or keys.
1. Physical The actual hardware and signals. Cables, radio, HSMs, TPMs, smart cards, secure boot firmware. Cryptography is embedded in hardware trust roots and secure device identity.

Why the model is useful even if it is simplified

Real systems do not stay neatly inside one layer. For example, a web application might rely on TLS at transport, JWTs at session/application level, and certificates in a hardware security module at physical level. The OSI model helps you sort these moving parts into a structure you can explain to others.

Section 4

Cryptography basics you need before talking about PQC

Encryption

Encryption turns readable data into unreadable data so that only authorized parties can recover it. The original readable data is called plaintext. The protected version is ciphertext.

Symmetric cryptography

The same secret key is used to encrypt and decrypt. It is fast and is often used for bulk data, such as TLS traffic after the handshake completes.

Asymmetric cryptography

A public key is shared widely, while a private key is kept secret. It is used for identity, key exchange, and signatures. Certificates are built on this model.

Hashing

A hash function turns data into a fixed-length fingerprint. Hashes are useful for integrity checks, password storage, and digital signatures. A hash is not encryption.

Digital signature

A signature proves that data came from the holder of a private key and has not been changed. It provides authenticity and integrity, not secrecy.

Certificate

A certificate is a signed document that binds a public key to an identity such as a domain name, server, organization, or person.

Why PQC matters

Post-quantum cryptography is the class of algorithms designed to resist attacks from quantum computers. A scanner is interested in where classical cryptography is used so the organization can plan which keys, certificates, and protocols would need replacement or migration.

The practical inventory question is not "is there any encryption at all?" The real question is: "where are the trust anchors, who owns them, how are they deployed, and what breaks if we change them?"

Section 5

Enterprise components, translated into beginner language

A large organization contains many kinds of systems. Some are obvious, like web servers. Others are easy to ignore, like DNS resolvers, HSMs, or proxy layers. For a scanner, the goal is to identify every place where cryptography is used directly or indirectly.

Application-layer components

  • Web applications: software that users open in a browser. They usually rely on TLS, cookies, sessions, and API calls.
  • APIs: program-to-program interfaces. They often use TLS, OAuth2, JWT, and mTLS for service identity.
  • SaaS applications: vendor-hosted services such as email, HR systems, or collaboration platforms. The scanner must watch SSO and admin credentials.
  • Load balancers and reverse proxies: traffic managers that sit in front of apps. They often terminate TLS, which means they hold or use certificates.
  • Message queues and caching systems: infrastructure that moves or stores data between app steps. They may not be "user-facing" but still carry secure connections.

Network-layer components

  • Routers: move traffic between networks.
  • Firewalls: decide which traffic can pass.
  • VPN gateways: create encrypted tunnels between sites or users and the organization.
  • DNS infrastructure: resolves names like example.com into IP addresses. Some DNS systems also validate signatures with DNSSEC.

Access and identity components

  • SSO systems: centralized login systems such as SAML or OIDC providers.
  • Certificate authorities and key management: systems that issue, store, rotate, and revoke keys and certificates.
  • Wi-Fi authentication and 802.1X: network access controls that can rely on certificates or RADIUS.
  • Hardware security devices: TPMs, HSMs, smart cards, and tokens that protect private keys from software exposure.
Section 6

How a web application actually works, step by step

Web applications are important because they are the most common place where cryptography, identity, and infrastructure overlap. A beginner should think of the flow as a chain of hops.

1. BrowserThe user types a URL or clicks a link. The browser decides which server to contact.
2. DNS lookupThe browser asks DNS where the domain lives. DNS is the Internet's phone book.
3. TLS handshakeThe browser and server agree on encryption and prove identity using certificates.
4. Web serverNginx, Apache, IIS, or a cloud edge service receives the request and may terminate TLS.
5. App serverThe application logic authenticates the user, checks permissions, and processes the request.
6. DatabaseThe app may read or write persistent data through a secure database connection.
7. Queue or cacheBackground jobs and cached data help the system stay responsive and reliable.
8. ResponseThe system sends data back through the same path, often with more than one cryptographic control in place.

What TLS is doing in that flow

TLS, or Transport Layer Security, protects data in transit. "In transit" means while it is moving over a network, not while it is stored on disk. TLS gives you three major properties:

  • Confidentiality: outsiders cannot read the traffic.
  • Integrity: outsiders cannot silently alter the traffic.
  • Authentication: the client can verify the server, and sometimes the server can verify the client.

Why the web path is not just "browser to server"

In real organizations, the browser often talks to a CDN or edge proxy first, then a load balancer, then a web server, then an application server, then one or more internal services. Each hop may use different certificates, different keys, and different trust assumptions. The scanner needs the full chain.

Section 7

Where these systems are hosted

Understanding location is important because cryptography changes depending on whether the system is on-premises, in a public cloud, inside a SaaS product, or running on endpoint hardware.

Hosting model What it means Typical crypto-related examples Scanner implication
On-premises Systems physically managed by the organization. Internal servers, firewalls, switches, HSMs, VPN gateways, directory servers. You can often inventory hardware, configs, and certificates directly.
Public cloud Infrastructure hosted by a cloud provider but controlled by the organization. Cloud load balancers, managed databases, cloud KMS, container platforms, edge services. The scanner must combine cloud APIs with workload inspection.
SaaS Software hosted and operated by a vendor. SSO integrations, admin portals, SCIM, certificates for federation, email signing. You may not see the server internals, so you scan the identity and integration surface.
Endpoint User devices and hardware close to the user. TPM, disk encryption, smart cards, FIDO tokens, secure boot, Wi-Fi authentication. These assets are critical because they often hold user identity and private keys.

Practical takeaway

A good scanner does not just ask, "Is there a certificate?" It asks, "Where is the certificate used, who owns it, where is the private key stored, how does it move across environments, and what systems depend on it?"

Section 8

What the scanner should scan first

The scanner should prioritize components by how likely they are to hold important cryptography and by how painful a migration would be if that cryptography changed.

High priority

Systems where certificates, key exchange, signatures, or trust anchors are direct and critical: TLS termination, VPN gateways, SSO providers, HSMs, TPMs, certificate stores, database encryption, and secure boot firmware.

Sample or defer

Systems that may use crypto but are not usually the primary migration bottleneck: caching layers, message queues, logging pipelines, proxies, and some internal libraries.

Usually out of scope

Systems with no meaningful cryptographic responsibility: pure routing devices without tunnel crypto, hubs, repeaters, clear-text VLAN tagging, and other components that only move data without securing it.

Why this ordering matters

Migration planning is limited by time, ownership, and risk. If you spend effort on low-value devices first, you may miss the assets that actually block PQC readiness.

How to explain the scanner logic in one sentence

"The scanner finds the places where trust is created, enforced, stored, or depended on, then ranks them by impact and migration difficulty."

Section 10

How to present this to other people

If you need to explain this in a meeting, do not start with abbreviations. Start with the problem: modern organizations depend on cryptography everywhere, but they often do not have a complete inventory of where it is used.

  1. Explain that the OSI model is just a map for organizing network concepts.
  2. Show that encryption is not only in web traffic, but also in identity systems, VPNs, Wi-Fi, databases, and hardware.
  3. Show that the scanner is trying to find trust points, not just encryptions.
  4. End with why PQC matters: organizations need to know what must change before quantum-safe migration becomes urgent.

One-sentence summary you can reuse

"This strategy maps every important cryptographic dependency in the enterprise so we can rank what must be migrated first for post-quantum readiness."
