SSL, TLS & SSH — Deep Internals
Every field the scanner reads, every term you will encounter
Why this matters SSL/TLS and SSH are where the majority of quantum-vulnerable cryptography lives at the network layer. The scanner reads specific fields from these protocols during the handshake phase. You need to understand every field it reads to know what it's reporting on.
SSL vs TLS — What's the Difference?

SSL (Secure Sockets Layer) is the predecessor to TLS (Transport Layer Security). SSL 2.0 and 3.0 are completely broken and were retired in 2011 and 2015. TLS 1.0 and 1.1 are deprecated. TLS 1.2 is the current minimum and TLS 1.3 is the modern standard. In everyday speech, people say "SSL" when they mean "TLS" — the HTTPS padlock in a browser is TLS, not SSL. Your scanner will report TLS version numbers — 1.0, 1.1, 1.2, 1.3 — and flag 1.0 and 1.1 as deprecated.

The TLS Handshake — Step by Step

The TLS handshake is the negotiation that happens before any application data is sent. It establishes: who the server is (authentication), what algorithms to use (negotiation), and a shared symmetric key for encrypting the session. The scanner reads and records data from steps 1–4 — it never needs to see the encrypted application data.

1
ClientHello (Client → Server)
Client opens TCP connection and sends ClientHello. Contains: TLS version supported, a random 32-byte nonce, a session ID (if resuming), and — critically — a list of all cipher suites the client supports, plus a list of named groups (key exchange curves/algorithms). The scanner sends a ClientHello advertising every cipher suite and named group that exists, including PQC ones like X25519MLKEM768, to see what the server picks.
Fields scanner reads: supported_versions, cipher_suites list, supported_groups (key_share extension)
2
ServerHello (Server → Client)
Server picks one cipher suite from the client's list and responds with ServerHello. This reveals: the negotiated TLS version, the selected cipher suite (e.g. TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384 or TLS_AES_256_GCM_SHA256 for TLS 1.3), and the server's chosen key share (the public portion of the key exchange). The scanner records all of this.
Fields scanner reads: version, cipher_suite, key_share → named group used (P-256, X25519, X25519MLKEM768...)
3
Certificate (Server → Client)
Server sends its X.509 certificate chain — the leaf certificate plus all intermediate CAs. Each certificate is a DER-encoded binary structure. The scanner parses every certificate in the chain to extract: subject name (who the cert is issued to), issuer name (who signed it), signature algorithm OID (is it RSA? ECDSA? ML-DSA?), public key algorithm and size, SANs (Subject Alternative Names — all domains covered), validity dates. The signature algorithm OID is the key PQC finding.
Fields: signatureAlgorithm (OID), subjectPublicKeyInfo (keyType + keySize), notBefore, notAfter, subjectAltName
4
ServerKeyExchange / CertificateVerify
In TLS 1.2: ServerKeyExchange carries the server's ephemeral public key (for ECDHE/DHE) and a signature over it. This signature uses the certificate's private key — if the cert is RSA, the signature is RSA. In TLS 1.3 this is merged into CertificateVerify. The scanner notes the ephemeral key exchange algorithm (ECDHE = vulnerable, ML-KEM hybrid = safe) from the named group in ServerHello's key_share extension.
Key exchange types: RSA (no forward secrecy, vuln), DHE (vuln), ECDHE (vuln), X25519MLKEM768 (hybrid, safe)
5
Finished — Encrypted Session Begins
Both sides confirm the handshake with a Finished message. From this point, all data is encrypted with the negotiated symmetric key (AES-256-GCM or ChaCha20-Poly1305). The scanner closes the connection here — it has everything it needs. No application data is read.
Symmetric cipher (AES-256-GCM, AES-128-GCM, ChaCha20) is safe against quantum — scanner notes it but doesn't flag it as vulnerable if AES-256

SSH Protocol — Internals

SSH (Secure Shell) is a protocol for encrypted remote access. Like TLS, it uses an asymmetric cryptography handshake (the KEX) to establish a symmetric session key, then uses symmetric crypto for the session. The scanner reads the SSH handshake similarly to TLS — by initiating a connection and reading the KEX Init messages.

CMDB & Cryptographic Asset Inventory
What a CMDB is, why scanning it is a competitive advantage, and how it feeds the scanner
What is a CMDB?

A CMDB (Configuration Management Database) is an organisation's central database of all IT assets — servers, applications, network devices, certificates, software components, cloud resources — and the relationships between them. It is the organisation's "single source of truth" for what exists in the IT environment: who owns it, what it runs, what it connects to, how critical it is, and what environment it lives in (production, staging, dev).

A CMDB stores Configuration Items (CIs) — each CI is one asset record. Every CI can have attributes (IP address, OS version, installed software, owning team, tier, location) and relationships to other CIs (this server hosts that application, that application depends on this database). This relationship graph is what makes a CMDB fundamentally different from a simple asset spreadsheet.

Why Scanning the CMDB Changes Everything — 7 Specific Advantages
The core shift Every other approach to PQC scanning starts blind — sweep an IP range, find what's listening, figure out what it is. CMDB integration flips this: you start with a complete, structured picture of what exists, and scan with context. The difference is not incremental — it changes the quality of every finding, every risk score, and every report the scanner produces.
Advantage 1 — Complete Target List With Zero Enumeration

Without a CMDB, the scanner sweeps IP ranges with Nmap, guesses at domains, and misses anything that isn't network-visible — internal services, shadow IT on odd ports, systems behind NAT. The scan coverage is whatever you thought to include in the IP range.

With CMDB access, you query for every CI in scope and get back a structured list of every server, every application, every service, every cloud resource — with hostnames, IPs, environments, and ports already known. The CMDB has already solved the discovery problem. You inherit that work instantly. Coverage goes from "what we pointed the scanner at" to "everything the organisation knows it runs."

# Without CMDB — blind sweep targets = nmap_sweep("10.0.0.0/8") # ~16 million IPs, hours to scan, misses HTTPS on port 8443 # With ServiceNow CMDB API cis = servicenow.query("cmdb_ci_server", filters={"environment": "production"}) # Returns: [{name, ip, hostname, os, env, tier, owner, services: [{port, protocol}]}] # Every server the org knows about — including the ones on port 8443, 9443, 4443
Advantage 2 — Risk Scoring Uses Real Business Context

The scanner's risk scorer needs two key inputs it cannot derive from a TLS handshake alone: data sensitivity (what kind of data flows through this system?) and exposure level (is this internet-facing or internal?). Without a CMDB, you use a default or ask the user to fill in a spreadsheet.

With CMDB access, these fields are already recorded. The CI record for a server says: Tier 1 production, internet-facing, PII data classification, owned by the Customer Identity team. The risk scorer pulls these directly and computes an accurate composite score — not an estimate based on port numbers and guesses.

Advantage 3 — Every Finding Has an Owner Immediately

The most common failure mode in vulnerability management is findings that sit unactioned because nobody knows who owns the affected system. A PQC scanner without CMDB integration produces: "RSA-2048 certificate found on 10.14.22.17:443." Someone now has to investigate what that IP is, what team manages it, and who is responsible for migrating it.

With CMDB access, the scanner enriches every finding at the moment of discovery: "RSA-2048 certificate found on api-prod-03 (10.14.22.17:443), owner: Platform Team, manager: Jane Smith, Tier 1 production, CI ID: SRV-00492." The finding goes directly into the right team's remediation queue with zero manual triage.

Advantage 4 — Dependency Graph Without Inference

CMDBs track relationships between CIs. Application A runs on Server B, which uses Database C, which connects to Cache D. These relationships are pre-built and maintained by the organisation. Without a CMDB, the scanner must infer these relationships from network traffic patterns or filesystem analysis — which is incomplete and slow.

With CMDB access, the scanner reads the dependency graph directly. When it finds a vulnerable certificate on Server B, it immediately knows Application A is affected — because the CMDB says so — without ever scanning Application A directly. The blast radius analysis ("if we migrate this certificate, what else is affected?") is answered by the CMDB's relationship model, not reverse-engineered from packet captures.

# Query ServiceNow: what applications depend on this server? relationships = servicenow.query("cmdb_rel_ci", filters={ "child": "SRV-00492", # server where we found the vulnerable cert "type": "Hosted On::Hosts" }) # Returns: [APP-1102 (user-portal-frontend), APP-1103 (identity-api), APP-1104 (audit-service)] # All three applications are now known to be impacted — without scanning any of them
Advantage 5 — Prioritisation by Business Impact, Not Just Algorithm Risk

An RSA-2048 certificate on a Tier 1 production system serving 10 million users is categorically more urgent than an identical certificate on a developer's test server. Technically they are the same finding — same algorithm, same vulnerability. But the business impact is incomparable.

Without CMDB context, both findings get the same risk score and appear in the same priority queue. With CMDB context, the Tier 1 production finding is automatically scored higher due to its business tier, data classification, traffic volume, and SLA. A CISO reading the priority queue sees the most business-critical systems at the top — not a random ordering based on which subnet was scanned first.

Advantage 6 — Software Version Cross-Referencing Without Scanning

CMDBs track software inventory — what version of OpenSSL, nginx, Java, or specific applications is installed on each server. This information is collected by CMDB discovery agents (ServiceNow Discovery, SCCM, Puppet facts) and stored as CI attributes. The scanner can cross-reference these version records against its algorithm registry before sending a single network packet.

"Server X has OpenSSL 1.1.1 installed" — from CMDB. Cross-reference: OpenSSL 1.1.1 does not support ML-KEM or ML-DSA. Flag as migration dependency. Zero network scanning needed. This is especially powerful at scale — instead of actively scanning 10,000 servers for their OpenSSL version, you query the CMDB once and get the answer for all of them instantly.

# CMDB software inventory query — no scanning needed software_cis = servicenow.query("cmdb_ci_spkg", filters={ "name": "OpenSSL", "version": {"startswith": "1.1"} # OpenSSL 1.1.x — no PQC support }) # Returns: list of all servers running OpenSSL 1.1.x — flag as PQC migration dependency # Same query works for BouncyCastle < 1.72, PyCryptodome < 3.19, etc.
Advantage 7 — Continuous Monitoring Becomes Cheap

Without a CMDB, re-running the scanner means re-scanning the entire IP range and all repositories from scratch. Every scan is full-coverage and expensive. You might run it monthly — which means a new vulnerable system deployed on Day 2 of the month isn't detected for 28 days.

With CMDB integration, the scanner watches the CMDB for changes — new CI added, OS updated, new service deployed, new application registered. Only the delta is rescanned. A new server provisioned this morning is scanned this afternoon, automatically, because the CMDB change event triggered a targeted scan job. Detection latency drops from weeks to hours.

The staleness problem — and how to handle it Most on-prem CMDBs have stale data. Assets get provisioned but not registered; CIs get decommissioned but not removed. The scanner should treat CMDB data as the primary but not exclusive source. Strategy: use CMDB as the high-confidence target list (scan everything the CMDB knows about), then run a lightweight network sweep (Nmap/masscan) of your IP ranges to catch anything the CMDB missed — shadow IT, unregistered instances, misconfigured systems. Findings from the sweep that have no matching CMDB CI are flagged as "unregistered asset" — a data quality alert for the CMDB team.

The 7 Infrastructure Layers
Every category of IT asset that contains cryptography — universal, no industry-specific context
Why 7 layers? Cryptography appears at every level of an IT stack. A scanner that only checks TLS endpoints misses 80% of the cryptographic surface. These 7 layers provide a systematic framework for ensuring nothing is missed. Each layer has different asset types, different scanning mechanisms, and different migration complexity.
Migration order matters L3 (HSM/Key Management) → L2 (PKI/Certificates) → L1 (Network/TLS) → L4 (Applications) → L5 (Data) → L6 (Infrastructure) → L7 (Endpoints/Third Party). You cannot migrate a higher layer until the foundational layer beneath it is ready. PKI must be migrated before TLS because TLS certificates are signed by the PKI. HSMs must be ready before PKI because root CA private keys live in HSMs.

The 5 Scan Mechanisms
How the scanner connects to systems — every method, every protocol, every credential required
No single mechanism covers everything Each mechanism covers different asset types. A production PQC scanner uses all five simultaneously, dispatched by a central orchestrator to a worker pool. The orchestrator decides which mechanism to use for each target based on the target type and available credentials.
M1 — Active Network Handshake

The scanner initiates a protocol handshake — TLS ClientHello, SSH KEX Init, IKEv2 SA Init, DNS query — and reads the server's response. The connection is aborted immediately after the scanner has what it needs. Zero production impact. No credentials needed.

M2 — Passive Network Tap / SPAN

The scanner listens on a mirrored network port (SPAN port, network tap, or inline sensor) and captures all traffic without sending any packets. TLS ClientHello and ServerHello are not encrypted — cipher suites and certificates are visible. This catches: actual negotiated ciphers in production (vs just what's configured), internal east-west traffic that active scanning might miss, and legacy clients that downgrade to weaker ciphers.

Phase 2 — Job Dispatch

The orchestrator takes the target list and creates scan jobs. Each job specifies exactly: target address, port, mechanism to use, and credentials (if needed). Jobs are pushed to a priority queue — internet-facing targets and high-risk assets get higher priority. A worker pool (Celery workers or ThreadPoolExecutor) pulls jobs and executes them concurrently. A rate limiter prevents flooding target networks.

# Job schema — what the orchestrator dispatches to each worker { "job_id": "uuid-v4", "target": "api.company.com", "port": 443, "mechanism": "M1_TLS_ACTIVE", "priority": 1, // 1=highest (internet-facing), 4=lowest (internal/low-risk) "credentials": null, // M1 needs no credentials "timeout_seconds": 10, "rate_limit_group": "network" // shared rate limit pool for network scans } # Rate limiter — prevents overwhelming target networks # Per-target limit: max 1 connection/5 seconds to same host # Global limit: max 100 concurrent outbound TLS connections # Jitter: random 0-500ms delay per job to avoid synchronized bursts
Phase 3 — Per-Job Execution

Each worker picks a job, executes the appropriate scanning mechanism, and returns raw data. This is the core scanning logic — five different code paths depending on mechanism type.

A
Mechanism execution → raw data
M1: TLS handshake → raw DER certificate bytes + cipher suite string + TLS version. M2: packet from SPAN → raw TLS record bytes. M3: SSH command output → file paths, config file text, command output strings. M4: API JSON response → key metadata dict. M5: AST traversal → list of (file, line, algorithm_call) tuples.
B
Parse raw data → structured objects
DER cert bytes → x509.load_der_x509_certificate() → Certificate object with all fields. Cipher suite string "TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384" → split into {kex: "ECDHE", auth: "RSA", cipher: "AES-256-GCM", mac: "SHA-384"}. API key dict {KeyAlgorithm: "RSA_2048"} → {algorithm: "RSA", key_size: 2048}.
cert.signature_algorithm_oid.dotted_string → "1.2.840.113549.1.1.11"
C
OID lookup → algorithm classification
Every cryptographic identifier (OID, algorithm name string, key type constant) is looked up in the scanner's algorithm registry — a curated database mapping identifiers to: algorithm family, specific variant, primitive type (signature/KEM/symmetric/hash), classical security level, PQC safety status, and NIST IR 8547 deprecation deadline. This is the core knowledge base of the scanner.
"1.2.840.113549.1.1.11" → { name: "RSA", variant: "RSA-SHA256", primitive: "signature", pqc_safe: false, deadline: "2035" }
D
Risk scoring → priority classification
Each finding is scored on 5 dimensions, each 1–5: (1) Algorithm risk — AES-256=1, AES-128=2, ECDSA=4, RSA=5. (2) Data sensitivity — public data=1, long-lived PII=5. (3) Exposure — air-gapped=1, internet-facing high-traffic=5. (4) Replaceability — config change=1, hardware replacement=5. (5) Regulatory obligation — no mandate=1, hard deadline=5. Sum (5–25) → Priority 1 (20–25, immediate), P2 (15–19, before 2028), P3 (10–14, before 2030), P4 (<10, before 2035).
E
CBOM record creation → persist to DB
Finding is normalised into a CycloneDX CBOM record (one of: Algorithm, Key, Certificate, Protocol). All required fields are populated. Record is written to the findings database with: scan_id, timestamp, job_id, target, mechanism, and the full CBOM record as JSON. The database is the single source of truth — all output formats are generated from it on demand.
Phase 4 — Aggregation & Dependency Graph

After all jobs for a scan run are complete, the aggregation phase processes all findings together to produce the unified output.

1
Deduplication
The same certificate may have been found by M1 (TLS scan), M3 (filesystem scan), and M4 (ADCS API). These are merged into one CBOM certificate record, enriched with context from all three sources. Dedup key: certificate serial number + issuer DN (for certs), or OID + location (for algorithms).
2
Dependency graph construction
Build the relationships: Root CA → Intermediate CA → TLS leaf cert → application using it. Library → algorithm it provides → application that calls it. HSM → key stored in it → cert that uses that key. This graph enables blast radius analysis: if you migrate the root CA's signing algorithm, what else breaks? The graph answer: every cert signed by it.
root_ca issuedBy → intermediate_ca issuedBy → api.company.com TLS cert uses → nginx uses → serves traffic from LoadBalancer
3
CBOM document assembly
All records and dependency relationships are assembled into a complete CycloneDX CBOM JSON document with the standard envelope: bomFormat, specVersion, serialNumber, metadata (scanner name/version, timestamp, scanned component), components array (all crypto asset records), dependencies array (all relationships).
4
Delta analysis (continuous scanning)
Compare current scan results to the previous scan stored in the DB. Identify: new findings (new vulnerable asset appeared), resolved findings (asset was migrated — PQC-safe algorithm now), changed findings (cert renewed but still with RSA), unchanged findings (still vulnerable, still there). Delta report shows migration progress over time — the key metric for stakeholder reporting.
