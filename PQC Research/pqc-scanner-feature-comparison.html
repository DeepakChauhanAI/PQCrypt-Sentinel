1. Purpose and Method

This guide compares the feature set of 15 PQC-relevant products (5 open source, 10 commercial) across 7 functional areas. The goal is to decide which features our tool should build, which to integrate from existing open source, and which to skip entirely. The last columns of each matrix mark implementation complexity for a new build:

  • Low = 1-2 weeks of focused engineering, well-documented techniques, off-the-shelf libraries available.
  • Medium = 1-2 months, requires integration work, multiple data sources, custom engineering.
  • High = 3-12+ months, deep domain expertise, novel research, or significant infra (HSM lab, runtime instrumentation framework, federated graph platform).

11. Implementation Complexity for Our Tool

Map of every feature category to engineering effort, prerequisites, and dependencies.

Low complexity (1-2 weeks each)

  • TLS active handshake scan - integrate pqcscan or sslyze.
  • SSH audit - integrate ssh-audit.
  • Hybrid PQC / downgrade detection - extension of the above.
  • CBOM output (CycloneDX) - use cyclonedx-python-lib or similar.
  • CT log monitoring - use crt.sh or Sectigo API.
  • HNDL / Mosca's Theorem model - the math is simple, the UX is the work.
  • Ticketing integration (Jira/ServiceNow) - REST webhooks.
  • Single docker-compose deploy - packaging exercise.
  • On-prem / air-gap deploy - same.

Medium complexity (1-2 months each)

  • VPN / IKEv2 / IPsec scan - ike-scan + custom parsing.
  • Passive network tap (SPAN) - Zeek or pyshark with PCAP.
  • CA inventory (ADCS, Vault, AWS PCA) - one connector per CA, ~1-2 weeks each.
  • SAST: source-code crypto API scan - Semgrep rules + QRAMM rules + custom crypto-API dictionary.
  • Dependency / SBOM scan - Trivy + CycloneDX ingestion.
  • Container image scan - Trivy or Grype wrapper.
  • Linux / Windows host agent (or SSH/WinRM remote exec) - choose one to start.
  • Kubernetes scan - K8s API client + ingress/secrets enumeration.
  • Cloud KMS inventory - one SDK per provider (AWS, Azure, GCP).
  • Database TDE scan - vendor-specific SQL queries.
  • CMDB sync (ServiceNow/NetBox) - bidirectional REST integration, schema discovery.
  • Multi-dim risk scoring -5 input factors, weighting, threshold tuning.
  • Migration progress tracking - scan-vs-scan diff engine.
  • Drift detection - scheduled re-scan + delta alerting.
  • Web dashboard / visualisation - React or simple HTML, a charting library.

High complexity (3-12+ months each, requires specialists)

  • Cert lifecycle automation + auto-renewal - ACME protocol, vendor hooks, change windows.
  • PQC / hybrid cert issuance - integration with EJBCA, Certdog, or in-house CA, requires HSM.
  • Binary / firmware deep analysis - disassemblers (Ghidra, Binary Ninja), custom crypto signature DB.
  • Runtime reachability - eBPF or dynamic instrumentation, per-language agents.
  • HSM enumeration (PKCS#11 / KMIP) - HSM vendor SDKs, lab access for testing, deep PKCS#11 knowledge.
  • Relationship graph / blast radius - graph DB (Neo4j), traversal engine, UI, performance tuning at scale.
  • Cipher translation / virtual patch - inline network proxy, latency-critical, crypto at line rate.
  • Multi-tenant SaaS - identity, billing, tenant isolation, customer-grade SLAs.

12. Recommended Build Set

MVP (8-12 weeks) - the "first useful scan in under an hour" goal

Goal: ship a single docker-compose stack that scans a customer's TLS/SSH endpoints, classifies algorithms, produces a CycloneDX CBOM, applies Mosca's Theorem, and shows a simple web dashboard. No agents, no CMDB sync, no cloud APIs. Pure network observation, plus optional CMDB CSV import.

  • TLS active scan (integrate pqcscan or sslyze)
  • SSH audit (integrate ssh-audit)
  • Hybrid PQC + downgrade detection
  • CycloneDX CBOM output
  • Multi-dim risk scoring (algorithm + exposure + data longevity)
  • HNDL / Mosca's Theorem per asset
  • Simple HTML/React dashboard (asset list, risk distribution, top-N remediation queue)
  • Single docker-compose deploy with Postgres
  • Optional: CMDB CSV import for ownership and relationship context

Phase 2 (months 3-6) - the "real inventory" expansion

Goal: pull in the data sources that turn "list of endpoints" into "complete crypto inventory of the organisation". This is where the from-scratch approach (CA queries, SBOM, cloud KMS) is integrated.

  • CA inventory: ADCS connector first, then Vault, then AWS Private CA
  • CT log monitoring (passive, public)
  • SBOM ingestion (CycloneDX/SPDX from build pipeline)
  • Container image scan (Trivy wrapper)
  • Cloud KMS inventory (start with AWS, add Azure/GCP in 4.x)
  • CMDB bidirectional sync (ServiceNow first, NetBox second)
  • Migration progress tracking (scan-over-scan diff)
  • Drift detection with alerting
  • Jira/ServiceNow ticketing integration

Phase 3 (months 6-12) - the "depth" features that nobody else does

Goal: cover the gaps that the matrix shows nobody has solved well. These are the moat features.

  • Vendor PQC readiness database (OpenSSL version -> ML-KEM support, Thales firmware -> PQC dates, AWS KMS -> hybrid cert support)
  • Relationship graph + blast radius view (Neo4j + traversal UI)
  • Linux/Windows host agent (or SSH/WinRM remote exec) for installed library enumeration
  • Kubernetes / etcd / secrets scan
  • Database TDE scan (MSSQL first, Oracle second)
  • SAST integration (Semgrep rules + QRAMM ingestion)
  • PQC downgrade nuance scoring (per-endpoint "PQC offered but downgrade allowed")

Phase 4+ (year 2+) - the "moat" features that justify a price tag

Goal: features that no commercial vendor does well and that differentiate a research-grade product.

  • HSM PKCS#11 / KMIP enumeration
  • Runtime reachability (eBPF or language-specific agents)
  • Binary / firmware crypto analysis (if your target is OT/IoT)
  • PQC / hybrid cert issuance (requires HSM investment)
  • Cert lifecycle automation
  • Cipher translation / virtual patch (network-edge only)
  • Multi-tenant SaaS tier

13. What to Integrate vs Build vs Skip

Integrate (1-2 days each, saves months)

  • pqcscan - Rust binary, ML-KEM + hybrid detection, structured JSON. Do not rebuild this.
  • SSLyze - Python, full TLS analysis. Battle-tested.
  • ssh-audit - Python, SSH config audit.
  • testssl.sh - Bash, comprehensive TLS grading.
  • Trivy - container + SBOM scanner with PQC-relevant detection.
  • CycloneDX Python lib - CBOM generation.
  • Semgrep + QRAMM rules - SAST crypto-API rules.
  • ike-scan - IKEv2/IPsec probing.
  • Zeek - passive network analysis.

Build (our differentiated value)

  • Risk scoring engine (multi-dim, per-asset)
  • Mosca's Theorem UI surface
  • Vendor PQC readiness database
  • Relationship graph and blast-radius view
  • Scan-over-scan progress tracking
  • CMDB bidirectional sync with relationship inference
  • Drift detection + alerting pipeline
  • Remediation roadmap generator
  • Dashboard and reporting

Skip (out of scope for v1, defer to Phase 4+)

  • HSM deep enumeration (high effort, niche buyer)
  • Runtime reachability (research-grade, hard to productise)
  • Binary/firmware analysis (Binarly has a strong lock here)
  • Cipher translation proxy (Palo Alto's territory, hardware-locked)
  • Multi-tenant SaaS (Phase 4+ once product-market fit is proven)

14. Decision Summary

  • 15 products reviewed: 5 open source, 10 commercial.
  • Best open source per category: pqcscan (network), QRAMM CryptoScan (SAST), IBM QSE + CBOMkit (CBOM), Trivy (containers), Semgrep (rules).
  • Best commercial reference architecture: CryptoNext COMPASS (open discovery + commercial probe + analytics) - study this structure.
  • Biggest gaps in the entire market: vendor PQC readiness DB, relationship/blast-radius view, mid-market deployment simplicity, runtime reachability.
  • MVP is achievable in 8-12 weeks by integrating existing open-source scanners and building a risk/CBOM/dashboard layer on top.
  • Phase 2 (months 3-6) is what makes the product a real platform: CA inventory, SBOM, cloud KMS, CMDB sync, drift detection.
  • Phase 3 (months 6-12) is what justifies commercial pricing: relationship graph, vendor DB, host agents, K8s, SAST, downgrade nuance.
  • Phase 4+ (year 2+) is the moat: HSM, runtime, binary, cipher translation - the things that justify a $200K+ price tag or a research-grade positioning.

Strategic positioning recommendation: the empty zone is "self-hosted, full-coverage, sub-hour deploy, $15K-$50K/yr mid-market". The MVP builds the floor of that zone; Phases 2-3 build the walls.
