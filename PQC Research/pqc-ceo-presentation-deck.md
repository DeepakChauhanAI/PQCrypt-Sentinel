## Slide 17: Where Encryption Lives: The 7 OSI Layers
* **Slide Title:** Scanning Across the 7 OSI Layers
* **Visual Idea:** A vertical stack of the 7 OSI layers, with highlights on Layer 7 (Apps), Layer 4 (TLS), and Layer 1 (HSMs/TPMs).
* **Key Messages:**
  - **Layer 7 (Application):** Scanning APIs, custom web portals, and database TDE (at-rest encryption).
  - **Layer 4 (Transport):** Scanning load balancers, service meshes, and web servers.
  - **Layer 1 (Physical):** Enumerating Hardware Security Modules (HSMs) and endpoint TPM chips.
* **Speaker Notes (1 Min):**
  > "Encryption lives at multiple layers of a corporate network. Our scan engine is built to look at everything: from the application code at Layer 7, down to the transport channels at Layer 4 like TLS, all the way down to physical hardware chips like HSMs and TPMs at Layer 1. A complete inventory must cover all these points."

## Slide 18: Hosting Models & Scanning Approaches
* **Slide Title:** Multi-Environment Scan Approaches
* **Visual Idea:** A matrix showing the approach for: On-Prem (Agent/SSH) | Private Cloud (APIs) | Public IaaS (Cloud APIs) | SaaS/SaaS Identity (Admin APIs).

* **Speaker Notes (1 Min):**
  > "Large organizations run in hybrid environments. Our scanner adapts its scanning strategy based on the hosting model. For cloud-native PaaS or SaaS, it queries cloud configuration APIs. For legacy on-premise hardware, it uses secure remote commands or local agents. This ensures we scan every asset type without requiring uniform host access."

## Slide 19: Prioritization Framework: What to Scan First
* **Slide Title:** Prioritizing Exposure & Risk
* **Visual Idea:** A hierarchy chart showing: (1) High Priority (Internet-facing TLS, PKI roots, HSM keys), (2) Defer (Internal DNS, tape backups), (3) Out of Scope (Physical security).
* **Key Messages:**
  - **Exposure Focus:** Internet-facing TLS and VPN gateways represent the highest HNDL risk and must be scanned first.
  - **Blast Radius Focus:** PKI root certificate authorities and KMS key vaults represent the highest security impact.
  - **Efficiency:** We sample-scan or defer low-risk areas (like internal DNS) to avoid scanning overhead.
* **Speaker Notes (1 Min):**
  > "To ensure efficiency, we follow a strict prioritization framework. We scan public, internet-facing systems and master key storage vaults first, as these are the most exposed and have the largest blast radius if compromised. We sample-scan or defer low-risk areas like internal backups, and keep physical security completely out of scope."

## Slide 20: Technical Approach: CMDB-First Discovery
* **Slide Title:** Leveraging Existing Asset Inventories
* **Visual Idea:** A workflow showing the scanner connecting to a CMDB (ServiceNow/NetBox), filtering for relevant assets, and targeting scans.
* **Key Messages:**
  - **Smart Target Selection:** Instead of scanning thousands of IPs blindly, the scanner queries the Configuration Management Database (CMDB).
  - **Eliminating Noise:** Filters out non-crypto assets (e.g., printers, basic network switches) using smart keywords and relationships.
  - **Efficient Operations:** Reduces scanning overhead, prevents network disruptions, and prioritizes critical servers first.
* **Speaker Notes (1 Min):**
  > "To make this technically feasible, our scanner is 'CMDB-First.' Instead of running slow, noisy network scans across everything—which can crash legacy systems—our software queries the organization's existing IT asset database. It automatically filters for the servers, databases, and services that handle keys, and targets those specifically."

## Slide 21: No-CMDB: 6-Phase Rollout Strategy
* **Slide Title:** Building the Inventory from Scratch
* **Visual Idea:** A staircase diagram showing the 6 phases: (1) Public TLS -> (2) CAs -> (3) Internal Net -> (4) KMS/HSMs -> (5) SBOMs -> (6) Code repos.
* **Key Messages:**
  - **Iterative Discovery:** Build the cryptographic inventory step-by-step.
  - **CA Enumeration:** Pulling from Certificate Authorities (like Active Directory CS) represents the largest source of internal cert truth.
  - **Reconciliation:** Deduplicates and maps relationships between endpoints, certificates, and application owners.
* **Speaker Notes (1 Min):**
  > "If a client has no CMDB, our scanner builds one from scratch in a 6-phase rollout. We start with public web scans, pull certs from internal CAs, probe the internal network, query key stores, analyze software packages, and run static scans on code. The engine then reconciles these overlapping lists into a single, unified database."

## Slide 23: Discovery Reconciliation & Deduplication
* **Slide Title:** Merging Conflicting Discovery Sources
* **Visual Idea:** A diagram showing raw data from Cloud APIs, Nmap, and CAs being matched by `(IP, Port, FQDN)` and outputting a single verified asset.
* **Key Messages:**
  - **Deduplication:** Merges identical endpoints seen by multiple discovery channels.
  - **Relationship Inferences:** Connects VMs to database services, applications to KMS keys, and certificates to CAs.
  - **Confidence Scoring:** Attaches confidence scores to each source, highlighting weak links for human review.
* **Speaker Notes (1 Min):**
  > "When the same system shows up in cloud APIs, a network scan, and active directory, our scanner automatically deduplicates the record using its IP, Port, and hostname. It links the VM to its database and KMS key, builds a dependency graph, and assigns a confidence score to each connection so the security team gets clean data."

## Slide 24: Active Scanning Probes
* **Slide Title:** Targeted Active Verification
* **Visual Idea:** Probes targeting: TLS Handshake (KEX group OID) | SSH KEX (host key checks) | SBOM (OpenSSL library checks) | SAST (hardcoded key checks).
* **Key Messages:**
  - **Active Handshakes:** Negotiates actual TLS and SSH connections requesting PQC keys (like ML-KEM).
  - **SBOM Analysis:** Parses package databases (dpkg, rpm) to check if vendor software bundles old, weak libraries.
  - **Config Audits:** Compares negotiated algorithms with server configuration files to flag hidden vulnerabilities.
* **Speaker Notes (1 Min):**
  > "Once assets are discovered, the scan engine verifies the encryption actually in use. It runs active handshake probes to see if the server supports post-quantum keys. It also parses software libraries (SBOM) to see if packages are using outdated versions of cryptographic libraries like OpenSSL. This active verification is what produces hard cryptographic evidence."
